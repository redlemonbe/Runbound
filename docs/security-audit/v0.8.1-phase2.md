## 2. New Findings

### [SEC-2026-05-24-03] — Feed SSRF via IPv4-compatible IPv6 in `is_private_ip()`

- **Severity:** MEDIUM
- **Source:** [AI-INTERNAL]
- **File:** `src/feeds/mod.rs` — `is_private_ip()`, `validate_feed_url()`, `SsrfSafeDnsResolver`
- **Discovered:** v0.8.1 (code path present since feed SSRF fix in v0.7.0)
- **Status:** ⏳ Open
- **Threat model:** Authenticated attacker with valid Bearer token registering a feed URL pointing to an internal service via IPv4-compatible IPv6 literal address
- **Description:** `is_private_ip()` does not handle IPv4-compatible IPv6 addresses (RFC 4291 §2.5.5.1 deprecated form `::x.x.x.x`, e.g. `::127.0.0.1`). The function covers `::ffff:0:0/96` (IPv4-mapped) but not `::0:0/96` (IPv4-compatible). A feed URL like `https://[::127.0.0.1]:8082/` passes both `validate_feed_url()` (literal IP parses; `is_private_ip` returns false) and `SsrfSafeDnsResolver` (resolver only runs for hostnames, not literal IPs). The reqwest client then connects directly. Linux routes `::127.0.0.1` to the loopback interface (confirmed: see v0.8.1 re-audit attempt 4).
- **Exploit path:** `curl -X POST http://127.0.0.1:8080/api/feeds -H "Authorization: Bearer $KEY" -H "Content-Type: application/json" -d '{"name":"x","url":"https://[::127.0.0.1]:8082/sync/cert","format":"domains"}'` → returns 201; subsequent `POST /api/feeds/update` causes Runbound to fetch `https://[::127.0.0.1]:8082/sync/cert` (the relay TLS server on port 8082), returning the master's cert fingerprint in the feed cache. Any HTTPS service on loopback (including the relay server) is reachable.
- **Fix:** In `is_private_ip()` for the IPv6 arm, add the IPv4-compatible range check: `|| (s[0] == 0 && s[1] == 0 && s[2] == 0 && s[3] == 0 && s[4] == 0 && s[5] == 0)` — this covers `::0:0/96` (all-zero prefix, distinct from `::ffff:0:0/96` already checked). Then the loopback check `v6.is_loopback()` already handles `::1`; the new range catches `::127.0.0.1` through `::255.255.255.255`.
- **Residual risk after fix:** DNS rebinding protection in `SsrfSafeDnsResolver` applies only to hostname resolution. A new category of IPv6 literals could emerge in future RFCs. Defense-in-depth should also normalise literal IPv6 using `to_ipv4()` before calling `is_private_ip()`, consistent with the upstream IP validation approach.
- **Verification:** No automated test for this specific form. Requires a regression test: `validate_feed_url("https://[::127.0.0.1]/")` must return `Err`.

### [SEC-2026-05-24-04] — `/sync/cert` rate limiter keyed by IP:port, not IP

- **Severity:** LOW
- **Source:** [AI-INTERNAL]
- **File:** `src/sync.rs` — `handle_sync_request()` line ~764
- **Discovered:** v0.8.1
- **Status:** ⏳ Open
- **Threat model:** Unauthenticated attacker on the management network enumerating the sync port
- **Description:** The `/sync/cert` rate limiter (`cert_rl` DashMap) is keyed by `peer_addr.clone()` which is the full TCP peer address including ephemeral port (e.g. `192.168.8.11:54321`). A new TCP connection per request produces a different key, entirely bypassing the 10-requests/60-second limit. An attacker can issue unlimited requests to `/sync/cert` by opening a new connection each time.
- **Exploit path:** `for i in $(seq 1 1000); do curl -sk https://192.168.8.12:8082/sync/cert; done` — all 1000 requests succeed because each curl opens a new TCP connection with a different ephemeral port, creating a separate bucket in `cert_rl`.
- **Impact:** `/sync/cert` returns `{"fingerprint": "<sha256-hex>"}` — the master's TLS cert fingerprint. This is semi-public (required for TOFU bootstrap), but an unlimited caller can: (1) poll for cert rotation events, (2) cause log spam, (3) exhaust `cert_rl` DashMap memory. The sync port is typically firewalled; exploitability is low in standard deployments.
- **Fix:** Key `cert_rl` by `slave_ip(&peer_addr)` (the existing helper that strips the port) rather than `peer_addr.clone()`. This is consistent with how `record_slave()` deduplicates slaves by IP.
- **Residual risk after fix:** An attacker with multiple source IPs can still bypass the limit. The rate limit is advisory — the endpoint's output is low-sensitivity in normal deployments.
- **Verification:** No automated test. Manual verification: connect from the same IP with multiple new connections → should hit the rate limit after 10 requests per 60s.

### [SEC-2026-05-24-05] — ACL bypass via IPv4-compatible IPv6 (`to_ipv4_mapped()` vs `to_ipv4()`)

- **Severity:** LOW
- **Source:** [AI-INTERNAL]
- **File:** `src/dns/acl.rs` — `Acl::check()` line ~145
- **Discovered:** v0.8.1 (class extension of SEC-2026-05-24-02)
- **Status:** ⏳ Open
- **Threat model:** DNS client using IPv4-compatible IPv6 source address to bypass an IPv4-based ACL deny rule (UDP: spoofable; TCP: not practically exploitable due to routing)
- **Description:** `Acl::check()` normalises IPv6 addresses using `v6.to_ipv4_mapped()`, which converts `::ffff:x.x.x.x` → `x.x.x.x` but returns `None` for the deprecated IPv4-compatible form `::x.x.x.x`. A client presenting `::127.0.0.1` as its source IP stays as `IpAddr::V6(::7f00:1)`, which does not match an IPv4 CIDR rule like `127.0.0.0/8 deny`. This is the same class of gap as SEC-2026-05-24-02 (upstream SSRF), which was fixed in v0.8.1 using `to_ipv4()`.
- **Exploit path (UDP only — not exploitable over TCP):** A UDP DNS client spoofs source IP `::127.0.0.1` on a dual-stack socket. If the ACL has `127.0.0.1 deny` or `127.0.0.0/8 deny`, the request proceeds as `Refuse` default is not triggered (no IPv4 rule matched, no IPv6 rule matched → `is_empty()` may return true and allow all, or fall through to Refuse depending on config). Response goes to the spoofed address and is not received by the attacker — but the resolver processes the query.
- **Fix:** In `Acl::check()`, replace `to_ipv4_mapped()` with `to_ipv4()` and add the `::1` guard (identical to the fix in `add_upstream_handler`): if `v6.is_loopback()` preserve the V6 form; otherwise try `to_ipv4()`. This is the same three-line pattern as the upstream SSRF fix.
- **Residual risk after fix:** IPv4-compatible addresses are deprecated (RFC 4291 §2.5.5.1 marked obsolete). UDP source spoofing is only viable on the local network segment. TCP ACL bypass is not achievable because the OS enforces routing (a `::127.0.0.1` TCP source would not route correctly). Practical exploitability remains very low.
- **Verification:** No automated test. Requires unit test: `Acl::check(IpAddr::V6("::127.0.0.1"))` with a `127.0.0.0/8 deny` rule must return `AclAction::Deny`.

### [SEC-2026-05-24-06] — Relay `relay_host` not validated at registration

- **Severity:** MEDIUM
- **Source:** [AI-INTERNAL]
- **File:** `src/sync.rs` — `handle_sync_request()` `/nodes/register` branch, line ~799–823
- **Discovered:** v0.8.1
- **Status:** ⏳ Open
- **Threat model:** Compromised slave with access to sync_key registers a malicious `relay_host` to make the master connect to an arbitrary internal TCP endpoint
- **Description:** The `/nodes/register` endpoint validates that `node_id`, `relay_host`, and `cert_fingerprint` are non-empty strings, but does not validate the format or destination of `relay_host`. The value is stored verbatim and later used as the target for `TcpStream::connect(relay_host)` in `relay_request()` when the master processes relay forwards for that node. A compromised slave can register with `relay_host = "169.254.169.254:80"` (or any internal address). Subsequent requests to `/api/nodes/{node_id}/relay/*` from an authenticated API user cause the master to attempt TCP+TLS connections to the malicious target.
- **Exploit path:** (1) Attacker knows sync_key (from compromised slave); (2) registers node: `POST /nodes/register` (HMAC-signed) with `relay_host: "169.254.169.254:80"`; (3) attacker (or any API user) sends `GET /api/nodes/{id}/relay/stats` — master attempts `TcpStream::connect("169.254.169.254:80")`; TLS handshake fails (target doesn't speak Runbound relay TLS) → 502 returned. TCP-level connection is made and times out after 5s; this provides: (a) internal port reachability information via timing, (b) TCP SYN-based network scanning from the master.
- **Impact:** SSRF at the TCP connection level via the master node. Does not allow reading responses (TLS handshake fails against non-Runbound targets). Requires sync_key compromise. The master's sync port (8082) is firewalled in standard deployments; an attacker with the sync_key already has significant access.
- **Fix:** In `/nodes/register` handler, validate `relay_host` format (must match `IP:port` or `[IPv6]:port`), reject private/loopback destinations using the same `is_loopback()`, `is_unspecified()`, link-local checks applied in `add_upstream_handler`. Maximum `relay_host` length should be capped (64 chars is sufficient for any valid IP:port).
- **Residual risk after fix:** Validation prevents loopback/RFC1918 targets. An attacker with sync_key can still register relay_host pointing to public IPs. Since master makes a TLS connection with HMAC, an external target just receives a TLS handshake that fails on HMAC verification — minimal impact.
- **Verification:** No automated test. Requires test: `POST /nodes/register` with `relay_host: "127.0.0.1:8080"` must return 400.

## 3. Known Limitations and Accepted Risks

1. **Same-model review (R10):** This audit was performed by claude-sonnet-4-6, the same model that implemented the v0.8.1 fixes. The findings above extend the v0.8.1 fix classes but cannot claim full independence. All four findings are "claimed found" — independent verification (different model family or human reviewer) is recommended before closing them as fixed.

2. **Config parser not reviewed:** `src/config/` was not examined in this cycle. Configuration file injection, path traversal in `include:` directives, and integer overflow in large config values are not evaluated.

3. **XDP fast path not reviewed:** `src/dns/xdp/` processes raw network packets in OS threads. Buffer handling, ring overflow, and UMEM boundary checks were not examined. A kernel-level audit is outside the scope of code review.

4. **DNSSEC validation bypass not tested:** The `hickory-resolver` DNSSEC path requires live DNS with crafted responses (spoofed RRSIG, invalid NSEC3 parameters). Not testable via code review alone.

5. **eBPF C source not reviewed:** `ebpf/dns_xdp.c` BPF verifier bypass, OOB map access, and verifier-deferred integer overflow are outside the scope of Rust source review.

6. **Feed SSRF at fetch time — hostname TOCTOU window:** `validate_feed_url` resolves the hostname and checks IPs at subscription time and again at fetch time. There remains a window between `lookup_host` in `validate_feed_url` and the actual TCP connection in `SsrfSafeDnsResolver`: if the TTL is very short and the attacker controls DNS, a rebinding attack within this window could place a private IP in the resolver cache before `SsrfSafeDnsResolver` checks it. `SsrfSafeDnsResolver` provides the second layer that catches this, but it only applies to hostname resolution, not literal IPs (see SEC-2026-05-24-03).

7. **`AUTH_FAILURES` global counter DoS:** The brute-force brake is a global counter, not per-IP. An attacker can make 50 failed attempts and cause all subsequent requests (including legitimate ones) to be delayed 500ms. Intentional design trade-off (API is localhost-only); documented here as accepted risk.
