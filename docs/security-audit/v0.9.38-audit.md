## Executive Summary

Cycle C audited 8 source files across the relay, ICMP, eBPF, and hasher subsystems added or significantly changed between v0.9.15 and v0.9.38. 8 new findings were identified. The most significant (SEC-C1, HIGH) is a partial regression of SEC-B1: the fix applied cert pinning to `relay_forward_handler` but left the config push path (`push_to_slaves`) using `NoCertVerifier` unconditionally. SEC-C8 (HIGH, Gemini) is a new finding: `IcmpStats::ban()` does not send a ban command to the XDP fast path, making API-triggered and flood-detected bans non-functional at the kernel level. Three Cycle B performance findings (PERF-1, PERF-3, PERF-5) were verified fixed. Four Cycle B security findings remain open.

This audit did not include fuzzing, static analysis tooling (cargo-audit, semgrep), or live network testing. All findings are static code review only.

## Security Findings

### SEC-C1 — Config push still uses NoCertVerifier (SEC-B1 partial fix regression)
**Severity:** HIGH  
**Source:** [AI-ADVERSARIAL: Claude Sonnet 4.6] — independently confirmed by [AI-ADVERSARIAL: Gemini 2.5 Pro]  
**File:** `src/api/relay.rs:314`  
**Discovered:** 2026-05-25  
**Status:** ⏳ Open  

**Threat model:** Network attacker with LAN access to the master-slave segment.

**Description:** The Cycle B fix for SEC-B1 (relay TLS cert verification bypass) applied cert pinning selectively: `relay_forward_handler` (line 235) uses `pinned_client_config(fp)` when a cert fingerprint is present. However, the config push function `push_to_slaves` (called for DNS zone replication, blacklist sync, and feed pushes) always calls `relay_tls_config()` which uses `NoCertVerifier`, regardless of whether a cert fingerprint is stored for the target slave.

**Exploit path:** ARP spoofing or route injection on the master-slave LAN → intercept TCP connection on sync port 8082 → present arbitrary TLS certificate → TLS handshake succeeds → intercept or modify DNS zone config push, injecting malicious zone entries into the slave.

**Fix:** Apply the same conditional in `push_to_slaves` as in `relay_forward_handler`: look up the slave's `cert_fingerprint` from the `SyncJournal` and use `pinned_client_config(fp)` when available.

**Residual risk after fix:** Nodes without a stored fingerprint (pre-v0.9.15 registrations) continue to use `NoCertVerifier` on config push. Forced re-registration of all slaves is required for full protection.

**Verification:** No automated test. Confirmed by code inspection comparing `push_to_slaves` and `relay_forward_handler` call sites. Recommend live test using mitmproxy on the master-slave segment.

### SEC-C2 — Relay-propagated ICMP bans bypass the in-memory ban DashMap
**Severity:** MEDIUM  
**Source:** [AI-ADVERSARIAL: Claude Sonnet 4.6]  
**File:** `src/sync.rs:1856`  
**Discovered:** 2026-05-25  
**Status:** ⏳ Open  

**Threat model:** Operator relying on `GET /api/icmp/stats` to audit active bans on slave nodes.

**Description:** When the master propagates an ICMP ban to a slave via relay, the slave's handler (sync.rs:1856) sends only to `ban_cmd_tx` (applying the ban to the BPF kernel map) but does not call `icmp_stats.ban(ip, BanSource::Relay)`. The slave's `IcmpStats.banned` DashMap is never updated. `GET /api/icmp/stats` on the slave returns an empty or incomplete `banned` array, even though the BPF-level ban is active. The `Relay` variant in `BanSource` was defined but is never used.

**Exploit path:** Not a direct exploit. The slave correctly blocks the IP at the XDP layer. However, an operator auditing ban state via the API on the slave is given incomplete information, which could delay incident response.

**Fix:** In the slave's relay ban handler, call `relay.icmp_stats.ban(ip_addr, BanSource::Relay)` alongside the existing `ban_cmd_tx.send(...)`.

**Residual risk:** None if fixed; logging inconsistency only if left open.

**Verification:** No automated test. Code review of sync.rs:1856. `BanSource::Relay` variant defined but no call site passes it to `ban()`, confirmed by grep.

### SEC-C3 — ICMP ban DashMap grows without bound
**Severity:** MEDIUM  
**Source:** [AI-ADVERSARIAL: Claude Sonnet 4.6] — independently confirmed by [AI-ADVERSARIAL: Gemini 2.5 Pro] (as PERF-C3)  
**File:** `src/icmp.rs:66`  
**Discovered:** 2026-05-25  
**Status:** ⏳ Open  

**Threat model:** Attacker generating ICMP flood from a large number of spoofed source IPs.

**Description:** `IcmpStats.banned` is a `DashMap` with no eviction policy and no TTL per entry. Each ban inserts an IP with an `Instant` timestamp and a `BanSource`. Manual unbans remove entries, but flood-detected bans are never removed automatically. In a sustained flood from many spoofed IPv4 addresses, the DashMap grows by one entry per unique source IP that crosses `ban_threshold`. At ~50 bytes per entry, 1M banned IPs consume ~50 MB. `banned_snapshot()` iterates the full map on every `/api/icmp/stats` call, degrading in proportion to map size.

**Exploit path:** Send ICMP packets from 10k+ spoofed IPv4 addresses at a rate sufficient to cross `ban_threshold` (default: 100 rate-limited packets/cycle). BPF maps absorb the kernel-side state; the userspace DashMap accumulates indefinitely.

**Fix:** Add a background cleanup task that removes ban entries older than a configurable TTL (e.g., 24h default). Alternatively, cap the DashMap at a maximum size and evict oldest entries.

**Residual risk:** Operator must choose a TTL that balances memory constraints against re-ban risk for persistent attackers.

**Verification:** No automated test. Code review of icmp.rs. Confirmed: no Drop impl, no background eviction task, no call to `retain()` or equivalent in the codebase.

### SEC-C4 — BPF rate-limit counter map (icmp_rl_counts) exhaustible via IP spoofing
**Severity:** MEDIUM  
**Source:** [AI-ADVERSARIAL: Gemini 2.5 Pro]  
**File:** `ebpf/dns_xdp.c:110`  
**Discovered:** 2026-05-25  
**Status:** ⏳ Open  

**Threat model:** Attacker with ability to spoof source IP addresses targeting the ICMP rate-limit tracker.

**Description:** The `icmp_rl_counts` BPF map is defined with `max_entries = 8192` (8k). The map tracks per-IP ICMP packet counts for flood detection. An attacker sending ICMP packets from 8192+ distinct spoofed source IPs exhausts the map. Once full, new source IPs cannot be tracked, effectively hiding the "real" attack IPs from the userspace flood detector. The `icmp_rate_limit` map (64k entries) is harder to exhaust but shares the same pattern.

**Exploit path:** Send ICMP packets from 8192+ spoofed IPv4 source IPs → `icmp_rl_counts` fills → subsequent packets from new spoofed IPs are no longer counted → flood detector cannot detect or ban real attack source.

**Fix:** Increase `max_entries` to match the `icmp_rate_limit` map (64k), or implement LRU eviction on the counter map. A probabilistic structure (count-min sketch) would provide flood detection without exhaustion risk.

**Residual risk:** Spoofed ICMP traffic is typically a nuisance, not a critical threat, since ICMP replies are also absorbed by BPF rate limiting.

**Verification:** No automated test. Code review of dns_xdp.c:110 confirming `max_entries = 8192` and absence of eviction policy.

### SEC-C5 — HMAC verification uses hand-rolled constant-time comparison
**Severity:** LOW  
**Source:** [AI-ADVERSARIAL: Gemini 2.5 Pro]  
**File:** `src/sync.rs:103`  
**Discovered:** 2026-05-25  
**Status:** ⚠️ Accepted risk  

**Description:** `hmac_verify_with_ts` implements its own constant-time comparison via a manual XOR-fold over a `zip` iterator, rather than using `subtle::ConstantTimeEq` directly on the full byte slices. The `zip` iterator terminates at the shorter of the two strings, which leaks timing information about length mismatches. For SHA256 hex output, both strings are always 64 characters, so the comparison is effectively constant-time for valid-length inputs. The final comparison uses `subtle::ct_eq` on a 1-byte accumulator.

**Exploit path:** For equal-length (64-char) signatures, all 64 bytes are always XOR'd — no timing leak on content. A different-length input fails the length check with potentially variable timing, but reveals only "your signature is not 64 characters" — which is not secret information.

**Acceptance rationale:** The implementation is correct for SHA256 hex outputs. The `subtle` crate is already a dependency and the final comparison uses it. Refactoring to `subtle::ConstantTimeEq` on the full slice would be cleaner but changes no security property for this specific use case.

**Verification:** No automated test. Code review of sync.rs:103. SHA256 hex is always 64 characters; the length-check path does not leak secret material.

### SEC-C6 — ICMP checksum update in eBPF (Gemini finding — Disputed)
**Severity:** HIGH (initial) → 🔄 Disputed  
**Source:** [AI-ADVERSARIAL: Gemini 2.5 Pro] — disputed by [AI-ADVERSARIAL: Claude Sonnet 4.6]  
**File:** `ebpf/dns_xdp.c:292`  
**Discovered:** 2026-05-25  
**Status:** 🔄 Disputed — not a bug  

**Gemini's claim:** `bpf_htons(ICMP_ECHO << 8)` produces `0x0008` on little-endian, not the required `0x0800`, causing corrupt checksums.

**Dispute:** `csum16_add` is documented as "endianness-agnostic." `icmp->checksum` is read by the CPU in host byte order (byte-swapped on x86). The `bpf_htons` call converts the addend to host byte order as well. The double byte-swap cancels:

```
network checksum 0xABCD → CPU reads as 0xCDAB
addend: bpf_htons(0x0800) = 0x0008
native add: 0xCDAB + 0x0008 = 0xCDB3
CPU writes 0xCDB3 → packet gets 0xB3CD

correct answer: 0xABCD + 0x0800 = 0xB3CD ✓
```

The implementation is correct. Gemini's finding is based on a misread of the endianness arithmetic.

**Verification:** Not verified by live test. The dispute resolution depends on the `csum16_add` implementation semantics. The ICMP responder is disabled by default (`enabled: false`). A live ICMP ping test through the XDP program on x86_64 would definitively close this dispute.

### SEC-C7 — Request body fully buffered before size check in relay handler
**Severity:** LOW  
**Source:** [AI-ADVERSARIAL: Claude Sonnet 4.6]  
**File:** `src/sync.rs:1567`  
**Discovered:** 2026-05-25  
**Status:** ⚠️ Accepted risk  

**Threat model:** Authenticated attacker with `sync_key`.

**Description:** `handle_relay_request` calls `req.collect().await` (buffering the full body into memory) before checking `body_bytes.len() > 65_536`. HMAC verification happens before body read, so unauthenticated callers cannot reach this path. However, an attacker with a valid `sync_key` could send a large chunked body (e.g., 512 MB with no Content-Length) causing full memory buffering before the 65 KB check fires.

**Exploit path:** Requires `sync_key`. Send POST /relay/dns with chunked encoding, no Content-Length, and a body larger than available RAM → OOM → process restart.

**Acceptance rationale:** Requires compromised `sync_key`. The relay port (8082) is intended for trusted LAN use only. Using `axum::body::to_bytes(body, 65_536)` instead of `req.collect().await` would fix this cleanly. Deferring to a future hardening pass.

**Verification:** No automated test. Code review of sync.rs:1567. Confirmed: `req.collect().await` called without a size limit before the 65KB check.

### SEC-C8 — IcmpStats::ban() does not send ban command to XDP fast path
**Severity:** HIGH  
**Source:** [AI-ADVERSARIAL: Gemini 2.5 Pro]  
**File:** `src/icmp.rs:80`  
**Discovered:** 2026-05-25  
**Status:** ⏳ Open  

**Threat model:** Attacker relying on incomplete ICMP ban enforcement; operator using the management API to ban IPs.

**Description:** `IcmpStats::ban()` only inserts the IP into the local `banned` DashMap (used for API reporting). It does not send an `IcmpBanCmd::Ban` to the XDP poll task via `ban_cmd_tx`, nor does it propagate the ban to slaves via `ban_propagate_tx`. As a result, IPs banned via the management API (`POST /api/icmp/ban`) or via the automated flood detector are never blocked in the XDP fast path. The ban appears in `GET /api/icmp/stats` but is not enforced at the kernel level.

**Exploit path:** Trigger the flood detector (or call `POST /api/icmp/ban`) for an attacking IP → IP appears as banned in the API → BPF fast path continues processing ICMP packets from that IP without restriction → flood-based denial of service continues unmitigated.

**Fix:** Update `IcmpStats::ban()` to send `IcmpBanCmd::Ban(ip)` to `ban_cmd_tx` and, when `ban_propagate_tx` is configured, send the IP to `ban_propagate_tx`.

**Residual risk:** Until fixed, ICMP flood mitigation above the BPF per-IP rate-limit layer is non-functional for API-triggered and flood-detector-triggered bans. The BPF per-IP rate-limit maps continue to function independently as a first-line defense.

**Verification:** No automated test. Code review of icmp.rs:80. Confirmed: no call to `ban_cmd_tx.send()` or `ban_propagate_tx.send()` originates from `IcmpStats::ban()`.

## Performance Findings — Cycle C

### PERF-C1 — Hash inconsistency between Rust hasher (CRC32c) and BPF hasher (FNV-1a)
**Impact:** HIGH  
**Source:** [AI-ADVERSARIAL: Gemini 2.5 Pro]  
**File:** `src/dns/hasher.rs` (Rust) and `ebpf/dns_xdp.c:166` (BPF)  
**Status:** ⏳ Open  

**Description:** The Rust domain hasher introduced in v0.9.38 uses CRC32c SSE4.2 on x86_64 (`src/dns/hasher.rs`), while the BPF XDP hasher uses FNV-1a (`dns_xdp.c:166`). A second inconsistency exists: the BPF hasher ASCII-lowercases the QNAME before hashing, while the Rust implementation does not. These divergences mean that queries for the same domain name compute different hash values in the BPF routing layer vs. the Rust worker layer, breaking per-domain CPU affinity (issue #67): queries for domain X may be routed to CPU Y by the BPF XDP loader but hashed to CPU Z by the Rust worker, negating cache locality.

**Fix:** Unify the hashing algorithm across both layers. Preferred: FNV-1a with consistent ASCII-lowercasing in both layers (BPF-friendly, no HW dependency, eliminates both inconsistencies). Ensure QNAME case normalization is applied identically in both layers.

**Note:** Gemini referenced `src/dns/hash.rs:11`; the current implementation is in `src/dns/hasher.rs`. The CRC32c implementation improves `LocalZoneSet` lookup throughput vs. SipHash; the inconsistency only affects per-domain CPU affinity routing.

### PERF-C2 — CPUMAP domain-routing limited to 64 cores
**Impact:** LOW  
**Source:** [AI-ADVERSARIAL: Gemini 2.5 Pro]  
**File:** `ebpf/dns_xdp.c:309`  
**Status:** ⚠️ Accepted risk  

**Description:** The domain-routing CPUMAP path guards with `if (cpu >= 64) { return XDP_DROP; }`, while the `CPUMAP` BPF map has `max_entries = 256`. On hosts with >64 cores, queries routed to CPUs 64–255 are dropped by this guard. The feature (`xdp-domain-routing: yes`) is disabled by default.

**Acceptance rationale:** Domain routing is an experimental feature off by default. The 64-core limit matches the vast majority of deployed hardware. Raising to 256 is a one-line fix when needed.

## Known Limitations and Accepted Risks

1. **No live network testing.** TLS interception (SEC-C1), ICMP responses (SEC-C6 dispute), and HMAC timing (SEC-C5) were evaluated by code review only. Live testing with mitmproxy and a packet analyzer would confirm or refute several findings.

2. **BPF verifier behavior not tested.** The analysis of the ICMP checksum (SEC-C6) relies on reasoning about `csum16_add` semantics. A live test sending ICMP pings through the XDP responder on an x86_64 host would definitively resolve the dispute.

3. **No fuzzing.** DNS wire format parsing, config parser, and serde deserialization in the relay handler were not fuzz-tested. Structured fuzzing with `cargo-fuzz` remains unperformed.

4. **Dependency audit not performed.** `cargo audit` was not run. SEC-A6 (unmaintained crates, accepted in Cycle A) was not re-evaluated.

5. **ICMP subsystem coverage incomplete.** The BPF-to-Rust channel design (ban_cmd_tx/rx, ban_propagate_tx) was reviewed at the call-site level only. End-to-end propagation was not traced through to the kernel BPF map operations. SEC-C8 may represent only part of the ban channel defects.

6. **Slave-side relay server not fully reviewed.** `handle_relay_request` in sync.rs was reviewed for body handling (SEC-C7) and HMAC (SEC-C5) but not for all operation handlers (`dns`, `blacklist`, `feeds`, `upstreams`).

7. **All open SEC-B findings remain unaddressed** (SEC-B7, SEC-B10, SEC-B13, SEC-B16). Operators should not rely on alert-block persistence or complete session cleanup for security guarantees.
