## 2. Verified Findings

### PERF-01 — Cache snapshot interval ✅ Fixed

**Prior claim:** `publish_loop` used `Duration::from_millis(100)` — new cache entries
were invisible to XDP workers for up to 100 ms.

**Verification:** `src/dns/cache_snapshot.rs:246`

```rust
let mut interval = tokio::time::interval(Duration::from_millis(10));
```

The interval is 10 ms. The double-buffer design (DashMap writer → ArcSwap<HashMap> reader)
ensures no reader-side blocking. The publish loop evicts expired entries before swapping.
**Status: Fixed. No action required.**

---

### PERF-02 — Mutex on mutable cache ✅ Fixed

**Prior claim:** `Mutex<HashMap>` caused contention above ~500 K insertions/s.

**Verification:** `src/dns/cache_snapshot.rs:63–65`

```rust
pub type CacheSnapshot       = HashMap<QuestionKey, CacheEntry>;
pub type SharedCacheSnapshot = Arc<ArcSwap<CacheSnapshot>>;
pub type MutableCacheMap     = Arc<DashMap<QuestionKey, CacheEntry>>;
```

`MutableCacheMap` is `Arc<DashMap<…>>`, not `Mutex<HashMap>`. DashMap uses sharded
RwLocks internally. The read-side snapshot uses `ArcSwap`, providing zero-lock reads
for XDP workers.
**Status: Fixed. No action required.**

---

### PERF-03 — No NUMA awareness ✅ Fixed

**Prior claim:** UMEM allocated without NUMA affinity; on dual-socket servers, XDP workers
could access remote NUMA memory.

**Verification:** `src/dns/xdp/umem.rs:538–565` — `rebind_to_local_numa()` calls
`mbind(MPOL_PREFERRED | MPOL_MF_MOVE)` to migrate pages to the worker thread's NUMA node.
The function is invoked from `src/dns/xdp/worker.rs:402`, immediately after CPU pinning:

```rust
// worker.rs:402
super::umem::rebind_to_local_numa(sock.umem.area, sock.umem.area_len);
```

Call ordering is correct: `pin_to_cpu()` sets the CPU affinity before `rebind_to_local_numa()`
queries the NUMA node via `SYS_getcpu` (vDSO). Any `mbind()` failure is silently absorbed —
the UMEM remains functional on non-NUMA or restricted environments.
**Status: Fixed. No action required on single-node systems.**

---

### PERF-04 — Hugepages optional ⚠️ Config gap

**Prior claim:** UMEM always allocated with 4 KiB pages; MAP_HUGETLB unavailable.

**Verification:** `src/dns/xdp/umem.rs:327–360` — `alloc_umem_area(size, hugepages: bool)`
attempts `MAP_HUGETLB | (21 << MAP_HUGE_SHIFT)` (2 MiB) when `hugepages=true`; falls back
to standard pages on `mmap()` failure. The `hugepages` flag originates from the
`xdp-hugepages: yes` configuration key and is passed through the UMEM constructor chain.

**Remaining gap:** This is a configuration gap, not a code gap. Hugepages require:
1. `xdp-hugepages: yes` in `runbound.conf`
2. `vm.nr_hugepages ≥ 512` in `/etc/sysctl.d/`

Neither is set by default. Deployments without hugepages incur additional TLB pressure
at high packet rates. Impact is workload-dependent; no measurement without bare-metal load.

**Status: Code correct. Operator configuration required before high-throughput deployment.**

---

### PERF-05 — No TX batching ✅ Fixed

**Prior claim:** Responses enqueued individually via `sock.tx.enqueue_tx(&[desc])`.

**Verification:** `src/dns/xdp/worker.rs:408–537` — the hot loop pre-allocates
`tx_descs: Vec<XdpDesc>` with capacity equal to the RX ring size, accumulates one
descriptor per processed packet, then calls `enqueue_tx(&tx_descs)` once per poll batch:

```rust
// worker.rs:411–412
let mut tx_descs: Vec<XdpDesc> = Vec::with_capacity(sock.rx.size as usize);

// worker.rs:536–537 (post-batch)
if !tx_descs.is_empty() {
    sock.tx.enqueue_tx(&tx_descs);
```

The kernel `sendto()` kick is issued at most once per batch (line 540), not once per frame.
**Status: Fixed. No action required.**

---

### PERF-06 — SO_REUSEPORT on UDP fallback ✅ Verified

**Prior claim:** Unverified whether SO_REUSEPORT was active on the Tokio UDP path.

**Verification:** `src/dns/server.rs:1384–1403` defines `bind_reuseport_udp()` which
creates a UDP socket with `SO_REUSEPORT` via the `socket2` crate, then binds it.
At `server.rs:1655–1670`, one socket per physical CPU is created when XDP is disabled:

```rust
// server.rs:1655
for _ in 0..ncpus {
    let udp = bind_reuseport_udp(&udp_addr)?;
    ...
}
```

The kernel load-balances incoming UDP datagrams across all bound sockets.
**Status: Verified. No action required.**

## 3. New Observations

### PERF-NEW-01 (LOW) — poll(1 ms) timeout adds up to 1 ms latency at idle

**Location:** `src/dns/xdp/worker.rs` — hot loop calls `poll(&mut pfd, 1, 1 /* ms */)`.

**Description:** At very low packet rates (less than one packet per millisecond per XDP queue),
a packet arriving just after `poll()` starts will wait up to 1 ms for the syscall to return.
At high load, `POLLIN` fires immediately so this is not on the critical path.

**Tradeoff:** The 1 ms sleep prevents 100% CPU usage at idle. A busy-loop variant would
eliminate this latency but consume a full core continuously. The current design is the
standard AF_XDP pattern for interactive-latency workloads.

**Status:** Accepted design tradeoff. No code change recommended.

---

### PERF-NEW-02 (INFO) — CPUMAP domain routing implemented

**Location:** `src/dns/xdp/loader.rs:18–130`.

**Description:** Issue #67 (CPUMAP-based per-domain CPU affinity) is fully implemented.
The XDP loader attempts the full BPF binary with `BPF_MAP_TYPE_CPUMAP`; if creation fails
(restricted `CAP_BPF`, kernel < 4.15, or slave VM), it automatically falls back to the
minimal binary without CPUMAP and logs a warning.

**Status:** Implemented. No action required.

## 4. Known Limitations

**KL-01 — [AI-INTERNAL] methodology only**
All verifications are static code review by the same model family as the implementation
authors. No independent human review or different model family has assessed these findings.
All "Fixed" verdicts are "code-structurally correct per static analysis" — not
runtime-validated.

**KL-02 — No runtime profiling**
No `perf`, `bpftrace`, or flamegraph data was collected. Bottleneck ranking is based on
code structure and established AF_XDP patterns, not measured data.

**KL-03 — No bare-metal benchmark**
Performance claims in `docs/performance.md` (v0.5.4, AMD Threadripper PRO 5995WX) have not
been re-run against v0.8.1. Throughput and latency numbers cannot be updated without a
new benchmark run on equivalent hardware. Any QPS estimates in this document would be
speculation; none are given.

**KL-04 — PERF-04 not validated under load**
The hugepages code path (`MAP_HUGETLB`) is structurally correct but has not been exercised
under load on the production hardware. TLB miss reduction from hugepages is workload-dependent
and cannot be quantified without measurement.

**KL-05 — NUMA migration not tested on multi-socket hardware**
`rebind_to_local_numa()` was verified correct by code review. Its effect (MPOL_MF_MOVE page
migration) on actual dual-socket EPYC/Xeon hardware has not been measured. Silent no-op
behaviour in restricted environments is confirmed by code inspection.

**KL-06 — poll() latency not measured at production packet rates**
The 1 ms poll timeout (PERF-NEW-01) adds latency only at idle. The threshold QPS below which
this matters has not been measured; on the production AMD 5995WX with 10+ XDP queues,
the per-queue packet rate at realistic load is expected to keep poll() returning immediately,
but this is not confirmed by data.
