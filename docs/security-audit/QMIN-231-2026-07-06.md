- **Lane A** — control-flow, termination, denial-of-service, panic/arithmetic safety.
- **Lane B** — DNSSEC integrity, cache poisoning, off-path spoofing, privacy.

### Severity scale

- **CRITICAL** — exploitable without authentication, or bypass of a documented guarantee.
- **HIGH** — authenticated exploit, silent corruption, or a practical single-source DoS.
- **MEDIUM** — reduces defence-in-depth; preconditions required.
- **LOW** — best-practice deviation; no direct exploit path.
- **INFO** — architectural observation.

No CRITICAL / HIGH / MEDIUM finding was identified. The DNSSEC fail-closed guarantee and the
off-path spoofing/poisoning resistance are preserved (see *Verified*). The two actionable
defects were introduced by the change itself and are fixed; the rest are accepted or disputed.

## Findings

### QMIN-1 — relaxed fallback latch was descent-wide, not cut-local — **LOW (privacy) — Fixed**

`descend()`'s `full` flag (fall back to the full QNAME after a probe misbehaves) was
initialised once per descent and never reset. A single lame/mute/hostile intermediate cut —
or a dead cached cut (#230) that forces a root restart — therefore disabled minimisation for
the **entire remaining descent**, leaking the full query name to every deeper cut (and, on a
root restart, to the root/TLD). This also contradicted the module/CHANGELOG claim that the
fallback is scoped to "that same cut".

- **Reported by:** Lane B (referral path) and maintainer verification (root-restart path) —
  independently, same root cause.
- **Impact:** privacy only. Integrity unaffected (answers are still DNSSEC-validated; leaked
  names only reach servers at/below the misbehaving cut). Not attacker-triggerable at will —
  the cut cache only stores cuts learned from successful descents.
- **Fix:** reset `full = false` (and `extra = 0`) on every accepted referral and on the
  root restart, making the fallback strictly cut-local. `src/dns/recursor_wire.rs`
  (referral block + `from_cache` restart).

### QMIN-2 — `MAX_DEPTH` shared between delegation depth and per-cut probing — **LOW (completeness) — Fixed**

Before #231, each iteration of the descent loop consumed a `MAX_DEPTH` slot only on a real
referral. With minimisation, empty-non-terminal probing (`extra += 1`) and the relaxed
fallback also consumed slots without advancing the cut, so a legal but pathologically deep /
flat name (≈20+ labels under one or two cuts) could exhaust `MAX_DEPTH = 24` and return a
spurious **SERVFAIL** for a name that resolves with minimisation off. Bounded failure — no
spin, no amplification (both auditors confirmed the loop is budget-bounded).

- **Reported by:** Lane A (F-1) and Lane B (F-3) — convergent.
- **Fix:** count only real delegation steps against `MAX_DEPTH` (separate `delegations`
  counter); the loop itself is hard-bounded by the query budget (`MAX_QUERIES = 80`, every
  iteration spends ≥1 upstream query). `src/dns/recursor_wire.rs`.

### QMIN-3 — query amplification per validated resolution — **LOW (DoS) — Accepted**

Minimisation raises the worst-case upstream-query constant per descent (~1 → up to
`1 + MAX_MINIMISE + 1 ≈ 12` per cut). Layered on the pre-existing #230 behaviour where the
DNSSEC chain walk resets `MAX_QUERIES` per `fetch`, the client-query→upstream-query ratio
grows. It remains **hard-bounded**: each `descend` caps at 80 queries via the shared
`&mut budget` (across `query_ns_set` and the mutually-recursive `resolve_ns_addr`), and the
number of fetches is bounded by `MAX_CNAME` × chain depth.

- **Reported by:** Lane A (F-2).
- **Decision — Accepted.** The absolute per-descent ceiling is unchanged and the vector
  predates #231 (#230). A single amplification budget threaded through the whole
  `resolve_validated` operation (instead of a per-fetch reset) is the correct hardening and
  is tracked as future work; `MAX_MINIMISE = 10` matches Unbound.

### QMIN-4 — `qname-minimisation` accepts any non-`"no"` token as ON — **INFO — Accepted**

`parser.rs` uses `val != "no"`, so a typo (`nope`, `0`, `false`) parses to ON, unlike sibling
booleans that use `== "yes"`. Chosen deliberately: the default is ON and the relaxed engine
never breaks resolution, so an ambiguous value **fails toward more privacy**, never toward a
silent disable. Documented here rather than changed.

- **Reported by:** Lane A (F-3). **Decision — Accepted** (fail-safe default).

### QMIN-5 — minimised probe question is low-entropy content — **INFO — Disputed (not a weakness)**

The first probe to a root server is a guessable `(TLD, A)`. This does **not** reduce off-path
spoofing resistance: the defending entropy is the random 16-bit transaction ID + the ephemeral
source port (connected UDP socket) + which of 13 root IPs was chosen — none of which
minimisation changes. The question content was never a secret (an attacker targeting a victim
name already knows it). Injecting a forged referral still requires guessing ID+port+server and
still yields a DNSSEC-validated answer downstream.

- **Reported by:** Lane B (F2). **Disputed** as a vulnerability; recorded for completeness.

## Verified (both lanes, with maintainer reproduction)

- **DNSSEC fail-closed preserved.** A minimised-probe response can never become the terminal
  message handed to the validator: every probing branch `continue`s; the terminal returns are
  guarded by `!probing`, so the validated message always carries the real `(qname, qtype)`.
  Live: `cloudflare.com → Secure`, `dnssec-failed.org → Bogus` (SERVFAIL), `google.com →
  Insecure`, `iana.org / verisign.com / isc.org / nic.cz → Secure`.
- **DS-at-parent anchoring (#230) intact.** DS/DNSKEY fetches are one label under their cut,
  so `minimised()` returns the real name immediately (`probing = false`) — they are never
  minimised to the wrong zone (no fail-open, no spurious Bogus). Locked by
  `minimised_is_noop_when_the_cut_is_the_direct_parent`.
- **No cache-poisoning widening.** The referral filter matches the real `qname`
  (`qname.is_in_zone(nm) && nm.is_in_zone(zone) && nm != zone`), not the probe prefix, so a
  lame/hostile in-bailiwick server gains no new delegation reach vs. the pre-#231 resolver.
- **Off-path spoofing resistance enforced for probes.** `id` + `question_matches(send_name,
  send_type)` are checked; the connected socket + per-query ephemeral port stand.
- **Termination & panic safety.** The loop is budget-bounded (≤ `MAX_QUERIES`); the sole
  `.expect()` in `minimised()` is unreachable (`zone` is always a suffix of `qname`, proven,
  so it strips ≤ `have-1` labels and never calls `parent()` on root). Budget is correctly
  shared through the mutually-recursive NS-address resolution.
- **Privacy goal met.** Packet capture of a cold resolution shows the root server receiving
  only `com. / org. / net. / one.` — never a full FQDN. NS-side and DNSSEC-chain fetches carry
  the NS name / the sensitive type only to the correct apex, never the victim's full name to
  the root (outside the QMIN-1 degradation, now fixed).
- **Serving path unchanged.** AD set only for `Secure` + DO=1; SERVFAIL on `Bogus` unless CD;
  no Bogus-as-valid path introduced.

## Method

Two adversarial AI auditors (disjoint lanes, no shared findings channel) read the branch diff
and the surrounding resolver/DNSSEC/cache code, each required to produce a concrete
failure/attack trace per finding (unreproduced findings discarded). The maintainer then
re-derived each finding against the source and, for QMIN-1/QMIN-2, reproduced the defect,
applied the fix, and re-ran the unit suite (467 passed), the live resolution + DNSSEC
Secure/Bogus/Insecure suite, the #230 cache regression, and the packet-capture privacy proof.
