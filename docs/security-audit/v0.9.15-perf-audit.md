# Runbound Performance Audit — v0.9.15

**Date:** 2026-05-25  
**Version audited:** v0.9.15  
**Source label:** [AI-INTERNAL] — Claude Sonnet 4.6 reading source code without live profiling data.  
**Goal:** 5M+ QPS target for state/enterprise proof-of-concept.  
**Methodology:** Static code review of hot-path files. No live profiling (`perf`, `flamegraph`, `bpftrace`) was performed. Findings are theoretical — actual bottlenecks must be validated with profiling on real hardware.

---

## Executive Summary

The XDP/AF_XDP fast path is architecturally sound for high-QPS targets. The primary throughput constraints under static analysis are: (1) a full DashMap traversal in the cache snapshot loop every 10ms, (2) double DashMap lookups in the rate limiter for new IPs, (3) per-query Vec allocations on the slow path, and (4) shared atomic hotspots on domain stats for popular domains.

Reaching 5M+ QPS requires the XDP fast path to handle the majority of queries (cache hit rate ~90%+). The slow path (hickory-server) is unlikely to exceed 500K QPS without the Vec allocation and lock contention issues addressed.

## Findings

### PERF-1 — Full DashMap clone every 10ms in cache snapshot loop
**Impact:** HIGH  
**File:** `src/dns/cache_snapshot.rs:276-284`

**Description:** The `publish_loop` task traverses the entire mutable DashMap every 10ms, cloning every key (`SmallVec<[u8; 64]>`) and every cached value (`Bytes + Instant`). Under a 5M QPS load with a large warm cache (~1M entries), this creates 100+ ms traversal spikes every 10ms — the lock contention from shard traversal can stall XDP worker threads that write cache entries concurrently.

**Recommended fix:** Use double-buffering: maintain a "delta" DashMap for new entries since the last snapshot. Every 10ms, atomically swap the delta into the published snapshot using ArcSwap. Only new/changed entries need to be cloned, not the full cache.

---

### PERF-2 — Double DashMap lookup in rate limiter; full retain() on exhaustion
**Impact:** HIGH  
**File:** `src/dns/ratelimit.rs:110-127`

**Description:** `check(ip)` performs `contains_key(&ip)` then `entry(ip).or_insert_with()` — two separate shard locks for every new IP. Under a slow-ramp DoS with 100k+ distinct source IPs, this serializes XDP workers. The `retain()` call on bucket exhaustion locks all DashMap shards simultaneously, causing a stop-the-world pause (estimated 10–100µs at 1M buckets).

**Recommended fix:** Replace the double-lookup with a single `entry().or_insert_with()`. Pre-allocate `DashMap::with_capacity(65536)` at startup. Consider a lockless probabilistic rate limiter (atomic CAS on a fixed-size array indexed by IP hash) as an alternative for the hot path.

---

### PERF-3 — Per-query Vec allocations on slow path
**Impact:** HIGH  
**File:** `src/dns/server.rs:692-725`

**Description:** Each forwarded query allocates multiple `Vec`s for wire-format encoding (`Vec::with_capacity(64)` for DNS name, `Vec::with_capacity(512)` for message). At 1M forwarded queries/sec (20% miss rate from 5M total), this is 2M+ Vec allocations/sec, driving ~1 GB/s of allocator pressure with the default system allocator.

**Recommended fix:**
1. Switch to jemalloc (low effort, likely +10-15% throughput): add `jemallocator` crate and `#[global_allocator]`.
2. Use `BytesMut` with pre-allocated thread-local buffers for wire encoding to eliminate per-query allocations entirely.

---

### PERF-4 — Atomic fetch_add contention on hot domain counters
**Impact:** HIGH (under adversarial hot-domain workload)  
**File:** `src/dns/xdp/worker.rs:590`, `src/dns/server.rs:876`

**Description:** `domain_stats.inc(&domain)` calls `fetch_add(1)` on a per-domain `AtomicU64`. Under 5M QPS with Zipf-distributed queries (top-10 domains getting 80% of traffic), each hot domain counter receives 400k+ atomic RMWs/sec. The MESI protocol causes cache-line invalidation loops across cores, degrading throughput as core count increases.

**Recommended fix:** Per-CPU/thread-local aggregation counters that are periodically merged (every 100ms) into the shared DashMap. Reduces atomic contention by the number of threads.

---

### PERF-5 — Per-packet rate-limit DashMap lookup in XDP hot path
**Impact:** HIGH  
**File:** `src/dns/xdp/worker.rs:573-577`

**Description:** Every received packet (even XDP fast-path cache hits) calls `rate_limiter.check(ip)` which acquires a DashMap shard lock. At 5M QPS on 16 AF_XDP queues, this is 312k lookups/sec per worker. Under high cardinality, multiple workers competing for the same IP's shard create serialization bottlenecks.

**Recommended fix:** Thread-local shadow cache with a small LRU (~1024 entries) for recently-seen IPs. Cache hit (99%+ of packets from known IPs) skips the DashMap entirely. Cache miss falls through to the global rate limiter.

---

### PERF-6 — QPS percentile recalculation on every snapshot request
**Impact:** MEDIUM  
**File:** `src/stats.rs:238-252`

**Description:** `percentile_ms()` iterates 13 histogram buckets per percentile on each API call to `/api/stats`. Negligible at low monitoring frequency, visible under continuous polling (dashboards). Pre-computed in snapshot would be cleaner.

**Recommended fix:** Pre-compute p50/p95/p99 in `qps_update_loop` (once per second) and store them in `StatsSnapshot`.

---

### PERF-7 — Blocking cache serialization on shutdown
**Impact:** MEDIUM  
**File:** `src/dns/cache_snapshot.rs:156-170`

**Description:** `save_xdp_cache()` traverses and serializes the full cache synchronously on shutdown. At 1M cached entries (~100 MB), this can block graceful shutdown for several seconds on slow disks.

**Recommended fix:** Offload to a background thread; add a 5-second timeout with log warning on failure.

---

### PERF-8 — String allocation inside logging on every query
**Impact:** MEDIUM (only at verbosity=INFO or higher)  
**File:** `src/dns/server.rs:312-320`

**Description:** `safe_name.to_string()` and timestamp formatting allocate a new String on every logged query. At INFO verbosity with 1M slow-path queries/sec, this drives 2M+ allocations/sec purely for logging.

**Recommended fix:** Use `tracing`'s `enabled!()` macro to guard all String allocations inside log callsites; move allocations inside the `if tracing::enabled!(...)` block.

## 5M QPS Feasibility Assessment

The XDP/AF_XDP architecture is the right approach. Modern Linux hosts can route 10–20M PPS through AF_XDP with zero-copy. The bottlenecks above are software constraints, not hardware limits.

**Path to 5M+ QPS:**
1. Cache hit rate must be ≥85% (XDP fast path is ~10x faster than slow path)
2. Fix PERF-1 (cache snapshot) and PERF-5 (per-packet rate limiter) to stop serializing XDP workers
3. Switch to jemalloc (PERF-3 partial fix, trivial effort)
4. Fix PERF-2 (rate limiter double-lookup) for adversarial IP cardinality scenarios

**No live benchmark data.** All findings are static analysis. Validation requires `perf stat`, `bpftrace`, or Runbound's own metrics under load from a packet generator (dnsmark, dnsperf, or pktgen).

---

## Known Limitations

1. **No profiling.** Static analysis cannot identify actual bottlenecks — only theoretical ones. A workload-driven profiling session with `cargo flamegraph` and a realistic DNS query distribution is required before optimization.

2. **eBPF program not analyzed.** `ebpf/dns_xdp.c` (the BPF program compiled at build time) was not reviewed. BPF verifier overhead and map operations in the kernel half of XDP may limit throughput independently.

3. **NUMA topology not considered.** The AF_XDP + tokio threadpool interaction under NUMA may produce unexpected affinity effects. This requires hardware measurement.

4. **jemalloc not tested.** The allocator switch (PERF-3) is estimated based on industry benchmarks, not measured on this workload.
