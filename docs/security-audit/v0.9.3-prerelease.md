## 2. Findings

### SEC-19 — WebUI lacks CSRF protection on POST endpoints [HIGH] [OPEN]

**Surface:** `src/webui/mod.rs` — `/login`, `/logout`, `/api/webui/password`

**Description:**
The WebUI POST handlers (login, logout, change-password) do not verify a CSRF token. The session cookie is `SameSite=Lax`, which provides partial CSRF protection for cross-origin navigations that use GET but not for POST forms submitted from the same site in some browser configurations.

More critically: `POST /api/webui/password` changes the admin password. An attacker who can trick the admin into visiting a malicious page could submit a forged password-change request if the browser's SameSite=Lax policy does not block it (depends on browser version and referrer policy).

**Severity:** HIGH — password change without CSRF token is a meaningful risk on management interfaces.

**Fix:** Add a double-submit CSRF token: on login success, generate a random 32-byte hex token, store in the session map alongside the session UUID, set it as a non-HttpOnly cookie `rb_csrf`. Verify `X-CSRF-Token` header matches on all state-changing endpoints. This is a standard pattern compatible with the current session architecture.

**Workaround:** Deploy WebUI behind an authenticated reverse proxy (e.g. nginx with `auth_basic`) for extra protection until fixed.

### SEC-20 — TSIG MAC comparison non-constant-time at Rust level [MEDIUM] [OPEN]

**Surface:** `src/dns/ddns.rs` — `tsig.algorithm.verify_mac()`

**Description:**
`TsigAlgorithm::verify_mac()` is a hickory-proto internal function. Its implementation uses `ring_like::hmac::verify()` which calls `ring`'s constant-time comparison (`ring::constant_time::verify_slices_are_equal`). This provides correct constant-time protection at the cryptographic level.

However, the key material is decoded from base64 on every request (`base64::decode(secret_b64)`) and the `tsig_keys` vector is scanned linearly. If the vector is long, a timing difference in key name lookup (string comparison) could be observable, leaking which key names are configured.

**Severity:** MEDIUM — practical exploitability requires very many keys (unlikely in practice) and microsecond timing resolution. Low real-world risk.

**Fix:** Cache decoded key bytes at startup (parse once, store `Vec<(String, TsigAlgorithm, Vec<u8>)>`); use constant-time comparison for key name lookup (e.g. `subtle::ConstantTimeEq`).

### SEC-21 — Serve-stale cache unbounded memory growth [MEDIUM] [OPEN]

**Surface:** `src/dns/server.rs` — `stale_cache: Option<Arc<DashMap<...>>>`

**Description:**
The stale cache stores one `Vec<Record>` per `(LowerName, RecordType)` key. Every successfully resolved domain populates an entry. There is no size cap: in a high-traffic resolver serving millions of distinct domains, the stale cache can grow to match the full forwarded query set.

For comparison, the XDP cache snapshot has a `cache_max_entries` cap; the hickory resolver has a `cache_size` from cgroup memory. The stale cache has no such bound.

**Severity:** MEDIUM — OOM risk under sustained high-QPS with diverse domains (e.g. a shared recursive resolver). Less severe on typical home/office deployments.

**Fix:** Cap stale cache at `cache_max_entries` (same parameter as XDP cache), using LRU eviction. Alternatively, reuse the hickory resolver's cache (which already enforces the cap) by inspecting it on SERVFAIL, avoiding a separate data structure.

### SEC-22 — Dynamic DNS records not persisted across restarts [MEDIUM] [DEFERRED v1.0]

**Surface:** `src/dns/ddns.rs` — `persist_dynamic_updates()` stub

**Description:**
DNS UPDATE records are applied in-memory via ArcSwap but not written to the persistent store (`src/store.rs`). A Runbound restart loses all dynamically added records. `ddclient` and similar tools will re-apply after the next TTL cycle, so the outage window is bounded by the update interval — typically 5 minutes.

**Severity:** MEDIUM — data loss on restart is expected for a v0.9.3 initial implementation but should be noted.

**Deferred:** Integration with `store::DnsEntry` planned for v1.0 (requires mapping `Record` → `DnsEntry` types and handling the schedule of dynamic updates from external clients).

### SEC-23 — Scheduled blocking relies on server UTC clock [LOW] [ACCEPTED]

**Surface:** `src/store.rs` — `ScheduleWindow::is_active_now()`

**Description:**
Schedule windows are evaluated against the server's UTC clock. Operators configuring schedules without understanding the UTC-only constraint may experience unexpected blocking/unblocking times. NTP drift ≤1s is typical and has no practical impact (schedule granularity is minutes).

Clock manipulation by a local privileged attacker (e.g., `date -s`) can shift schedule windows. This is an inherent limitation of any time-based policy system.

**Severity:** LOW — accepted as a known limitation. Timezone support is out of scope for v0.9.3.

**Accepted risk:** Documented in configuration reference.

---

### SEC-24 — Stale DNSSEC state visible after upstream recovery [LOW] [ACCEPTED]

**Surface:** `src/upstreams.rs` — `dnssec_stripping` field

**Description:**
`dnssec_stripping` is set to `true` when a probe returns `Some(false)` after a prior `Some(true)`. It is cleared when the probe returns `Some(true)` again. During a brief upstream instability where DNSSEC validation toggles, the field may oscillate, generating spurious stripping warnings in logs.

**Severity:** LOW — log noise only; no security impact beyond potential alert fatigue.

**Accepted risk:** Alert hysteresis (e.g., require 2 consecutive false AD-bit probes before flagging) could reduce noise but adds code complexity. Accepted as-is for v0.9.3.

---

### SEC-25 — WebUI session tokens not invalidated on password change [INFO] [OPEN]

**Surface:** `src/webui/mod.rs` — `change_password` handler

**Description:**
When the admin changes the password via `POST /api/webui/password`, existing session tokens in the `sessions` DashMap are not invalidated. An attacker who has stolen a session cookie retains access until the 8-hour TTL expires.

**Severity:** INFO — password change is typically an admin-only action; session stealing requires a prior compromise. Standard web security practice is to invalidate sessions on credential change.

**Fix:** Clear `state.sessions` on successful password change.

---

### SEC-26 — QPS sparkline SSE stream held open by browser [INFO] [OPEN]

**Surface:** `examples/web-ui/index.html` — `initSparkline()` / `fetch(BASE + '/stats/stream')`

**Description:**
The sparkline opens a long-lived SSE connection via `fetch()` with no reconnect backoff. If the server restarts, the browser will retry immediately (fetch Promise rejects → error swallowed → connection dropped). No throttling of reconnect attempts is implemented.

**Severity:** INFO — minor; browser will not hammer the server because the fetch rejects immediately and the error handler does nothing (silent catch). No reconnect loop is started.

**Fix:** Add exponential backoff on reconnect for robustness. Not security-critical.

## 3. Performance Analysis

### WebUI authentication overhead
- argon2id hash on login: ~80ms on modern hardware (by design — deliberate work factor). No concern.
- Proxy path: one `reqwest` request per API call with header injection. Overhead: <1ms per request. Negligible.
- Session DashMap lookup: O(1) amortized. No concern.

### Serve-stale
- DashMap insert per successful forward: 1 lock acquisition per domain. At 50k QPS with 100% cache miss rate, this adds ~0.1ms contention per second distributed across shards. Negligible.
- SERVFAIL path with stale hit: DashMap lookup + Vec clone + ArcSwap store. Total added latency: ~10µs. Acceptable.

### RFC 2136 Dynamic DNS
- TSIG MAC computation: HMAC-SHA256 per UPDATE. At typical DDNS rates (1 update/5min), negligible.
- base64 decode per UPDATE: eliminated by caching key bytes (see SEC-20 fix recommendation).

### Scheduled blocking
- `schedule_enforce_loop` wakes every 60s, loads blacklist from disk, rebuilds zone set. Cost: ~1ms per execution. Negligible.
- `is_active_now()` per blacklist entry at each DNS request: only called during `build_zone_set()` (reload/enforce loop), not on the hot query path. No per-query overhead.

### DNSSEC stripping detection
- Added one field write per probe result. Probes run every 30s. Zero query-path overhead.
