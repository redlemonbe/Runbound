**Key adversarial observations:**

  zones that were statically configured in `unbound.conf`. A TSIG-authenticated attacker

  (e.g., `*.doh.cloudflare.com`) can saturate the cache faster than the internal audit
  estimated.

## 2. New Findings

### SEC-AGV-01 — DDNS UPDATE overwrites statically configured zones [HIGH] [OPEN]

**Surface:** `src/dns/ddns.rs` — `handle_update()`, zone application logic

**Description:**
The UPDATE handler applies records to `LocalZoneSet` without checking whether the
target zone name was dynamically created or statically configured. At lines 141–145
of `ddns.rs`:

```rust
new_zones.zones.entry(n.clone()).or_insert(ZoneAction::Static);
new_zones.records.entry(n).or_default().push(rr.clone());
```

`or_insert(ZoneAction::Static)` only inserts if the name is absent. If `n` already
exists (because it was configured in `unbound.conf`), its existing records are left
in place and the new record is appended. This is the intended RFC 2136 ADD semantics.

However, the DELETE operations (`class=ANY(255)`, `class=NONE(254)`) apply equally
to statically configured names:

```rust
(RecordType::ANY, 255) => {
    if let Ok(n) = Name::from_ascii(name_str) {
        new_zones.records.remove(&n);   // removes static records
        deleted += 1;
    }
}
```

An attacker with a valid TSIG key can:
1. Send `DELETE ALL` for any static zone name (e.g., `router.local.`, `runbound.local.`)
2. Add spoofed A/AAAA records pointing to an attacker-controlled IP
3. All clients that query the resolver for that name are redirected

This bypasses the intent of `unbound.conf` static zones, which are expected to be
immutable at runtime. The impact depends on what names are in the local zone set,
but in typical deployments these include DHCP-assigned hostnames and management IPs.

**Severity:** HIGH — requires valid TSIG key, but TSIG keys are long-lived shared
secrets; compromise of the key material gives persistent name hijacking capability
for any local zone.

**Fix:** Maintain a `static_names: HashSet<Name>` in `LocalZoneSet` populated at
startup from `unbound.conf` entries. In `handle_update()`, reject DELETE operations
targeting names in `static_names` with `ResponseCode::Refused`.

### SEC-AGV-02 — Malformed schedule window silently disables blocking rule [MEDIUM] [OPEN]

**Surface:** `src/store.rs` — `ScheduleWindow::is_active_now()`

**Description:**
`is_active_now()` parses `start` and `end` strings with:

```rust
let Ok(sh) = self.start[..2].parse::<u8>() else { return false; };
```

A malformed schedule (e.g., `start: "99:00"`, `end: "ab:cd"`, or a string shorter
than 5 characters causing a panic on `[3..5]`) causes the method to return `false`
rather than returning an error. The consequence is:

- A parental-control blocking rule with a typo in the schedule (`"24:00"` intended
  as "end of day", which is invalid) will never apply: `is_active_now()` always
  returns `false`, so the domain is never blocked during the intended window.
- Worse: on `build_zone_set()` and `schedule_enforce_loop()`, the domain is also
  never added to the active zone set even when the schedule should be active.
- Panics: `self.start[..2]` will panic if `start` has fewer than 2 bytes. Malformed
  input from a trusted API consumer crashes the enforcement loop.

**Severity:** MEDIUM — not a remote exploit, but a content-filter bypass via misconfiguration.
The panic risk requires an authenticated API consumer to submit the bad value.

**Fix:**
1. Validate `start` and `end` on write in the API handler (`parse::<u8>()` + range checks).
2. In `is_active_now()`, use `get(..2)` / `get(3..5)` (returns `None` on short string) to
   avoid panics; return `Err` or a validated type rather than silently returning `false`.

### SEC-AGV-03 — Stale cache serves records across zone authority changes [LOW] [OPEN]

**Surface:** `src/dns/server.rs` — stale cache lookup, `serve_stale` path

**Description:**
When an operator removes a domain from the local zone set (e.g., deletes a blacklist
entry or removes a static zone), the stale cache retains the old record and will serve
it to clients who hit SERVFAIL within `stale_max_age` seconds.

This means a domain that was unblocked by deleting its blacklist entry can still
return NXDOMAIN or a sinkhole IP from the stale cache for up to 86,400 seconds (the
default `stale_max_age`) if the upstream resolver fails during that window.

**Severity:** LOW — requires simultaneous upstream failure, and the operator action to
remove the block. In practice the stale cache is only consulted on SERVFAIL, not on
a healthy resolution. The window is bounded by `stale_max_age`.

**Fix:** When a blacklist entry is removed or a zone is deleted, evict the corresponding
key from the stale cache. Alternatively, check the current zone set before serving a
stale record and skip stale serving if the name is in the blacklist.

### SEC-AGV-04 — WebUI proxied API requests include Authorization header in logs [INFO] [OPEN]

**Surface:** `src/webui/mod.rs` — proxy handler, `reqwest` client

**Description:**
The WebUI proxy injects the master API key as `Authorization: Bearer <key>` into
proxied requests. If `tracing` is configured at `DEBUG` or `TRACE` level, the
`reqwest` library may log full request headers including the Authorization value.

The default Runbound log level is `INFO`, so this is benign in default configuration.
At `DEBUG` level — which operators sometimes enable during troubleshooting — the API key
appears in journald logs accessible to any local user who can run `journalctl`.

**Severity:** INFO — requires DEBUG log level and local read access to journald.

**Fix:** Add a note to the debug-logging documentation that `RUST_LOG=debug` exposes
the API key in logs when WebUI proxy is active. No code change required.

## 3. Corroborated Findings (from [AI-INTERNAL])

### SEC-19 — CSRF on WebUI POST endpoints [HIGH] — CONFIRMED

The [AI-INTERNAL] analysis is correct. `SameSite=Lax` does not protect POST requests
submitted via `<form method=POST>` from a cross-origin page on the same registrable
domain (e.g., an attacker-controlled `attacker.runbound.local` if the admin has
configured that name). The double-submit cookie pattern is the correct fix.

### SEC-20 — TSIG key scan timing [MEDIUM] — CONFIRMED, SEVERITY MAINTAINED

Confirmed as written. A TSIG deployment with >10 keys would exhibit measurable timing
differences on slow hardware. The base64 decode per-request is an unnecessary cost;
caching at startup is the correct fix.

### SEC-21 — Stale cache OOM [MEDIUM] — CONFIRMED, SEVERITY ESCALATED

The internal audit estimated 50k QPS. At 200k QPS (a reasonable peak for a shared
office resolver), the stale cache insertion rate is 4× higher. With large record sets
(e.g., MX + SPF + DKIM TXT per domain), each entry can be 2–4 KB. 10M distinct
domains → ~20–40 GB heap before the OOM killer fires. LRU cap at `cache_max_entries`
is necessary for multi-tenant deployments.

## 4. Disputed Items

### SEC-24 — DNSSEC stripping oscillation — DISPUTE: should be MEDIUM, not LOW

The [AI-INTERNAL] audit accepted SEC-24 at LOW with "log noise only; no security
impact beyond alert fatigue." This adversarial review disputes the severity.

**Reason:** In a deployment with automated alerting on DNSSEC stripping events,
oscillation caused by flaky upstream validation (e.g., a misconfigured upstream that
returns AD=0 on every other response for a subset of zones) will trigger repeated
WARN-level log events. If SIEM rules count stripping events and alert at threshold,
oscillation can flood and suppress the SIEM alert queue — causing a real DNSSEC
stripping event to be lost in the noise.

**Recommended action:** Implement hysteresis (require 2 consecutive AD=0 probes before
setting `dnssec_stripping = true`). Low-effort fix, removes the dispute.
