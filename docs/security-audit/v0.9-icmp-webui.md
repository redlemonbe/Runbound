**Findings: 7 total** — 0 CRITICAL, 0 HIGH, 2 MEDIUM (SEC-12, SEC-16), 2 LOW (SEC-13, SEC-14), 1 LOW/ACCEPTED (SEC-15), 2 INFO (SEC-17, SEC-18).

**Key observations:**
- BPF bounds checks prevent memory safety issues in all cases, but incorrect parsing with IP options is functionally wrong (SEC-12)
- The `burst` config parameter is documented and user-facing but has no effect in BPF code (SEC-13)
- WebUI proxy buffers up to 8 MB before forwarding, while API backend limit is smaller — memory spike under concurrent large uploads (SEC-16)
- Rate limiting works correctly for normal traffic; fixed-window and LRU eviction limitations are known tradeoffs (SEC-14, SEC-15)

## 2. Findings

### SEC-12 — IHL not validated before ICMP header offset [MEDIUM] [OPEN]

**Surface:** `ebpf/dns_xdp.c`, ICMP handler branch

**Description:**
The ICMP handler computes the ICMP header pointer as `(void *)ip + 20`, assuming a standard
IPv4 header with no options (IHL=5). However, the `ip->ihl` field is not checked in the
ICMP branch. The IHL check (`if ((ip->ihl & 0xF) != 5)`) only appears later in the UDP branch.

```c
// ICMP branch — no IHL check
struct icmphdr *icmp = (struct icmphdr *)((void *)ip + 20);
if ((void *)(icmp + 1) > data_end)
    return XDP_PASS;
if (icmp->type != ICMP_ECHO)
    return XDP_PASS;  // non-echo ICMP → kernel
```

A packet with IHL=6 (4 bytes of options) has its ICMP header starting at offset 24, not 20.
The handler would read `ip->options[0..3]` as the ICMP type/code/checksum. If those bytes
happen to equal `type=8` (ICMP_ECHO), the handler would:
1. Accept a non-ICMP packet as an echo request
2. Swap MACs/IPs and change a byte in the original options field
3. Return XDP_TX — sending a corrupted packet back to the source

**BPF memory safety:** The bounds check `(void *)(icmp + 1) > data_end` prevents out-of-bounds
reads regardless of IHL. The BPF verifier accepts this. No kernel memory safety issue.

**Exploitability:** An attacker on the same LAN can craft an ICMP-with-options packet (IHL=6).
With XDP in SKB mode (current deployment), the NIC driver receives it and passes it to XDP.
Impact is limited: the mangled packet is returned to the attacker with a corrupted byte in
their own options field. No privilege escalation, no amplification beyond the original packet.
IP options are stripped by most internet routers — this affects LAN attackers primarily.

**Fix:** Add `if ((ip->ihl & 0xF) != 5) return XDP_PASS;` at the start of the ICMP branch,
before computing the ICMP header offset. Mirrors the existing check in the UDP branch.

### SEC-13 — `burst` config parameter has no effect in BPF rate limiter [LOW] [OPEN — behaviour bug]

**Surface:** `ebpf/dns_xdp.c`, rate limit check; `src/icmp.rs`, `src/config/parser.rs`

**Description:**
The `icmp_cfg_entry` struct in BPF and the `IcmpCfgEntry` struct in Rust both carry a `burst`
field. The config parser exposes `rate-limit-burst` in `runbound.conf`, and the REST API
accepts and returns `burst` in `PUT /api/icmp/config`. However, the BPF code never reads
`cfg->burst`:

```c
if (r->count >= cfg->rate_pps) {  // only rate_pps is checked
    // rate_limited
}
```

The `burst` parameter is silently ignored. Users who set `rate-limit-burst: 20` in config
will not get the expected token-bucket burst behavior — the effective burst is always 1 (the
first packet in a new window).

**Security impact:** None directly. Operators may configure a burst expecting it to be honored,
then observe higher drop rates than intended and conclude ICMP is not working.

**Fix:** Implement burst in BPF: `if (r->count >= cfg->burst + cfg->rate_pps * window_fraction)`
or simpler, allow the first `burst` packets unconditionally before applying `rate_pps` checks.

### SEC-14 — Fixed-window rate limiter: 2× burst at window boundary [LOW] [ACCEPTED]

**Surface:** `ebpf/dns_xdp.c`, `icmp_rate_entry` / window logic

**Description:**
The rate limiter uses a 1-second fixed window starting at the first packet timestamp. An
attacker sending `rate_pps` packets at the end of window T (e.g. t=0.99s) and `rate_pps`
packets at the start of window T+1 (t=1.00s) can pass 2× `rate_pps` packets within a ~20ms
span.

**Accepted risk:** This is a known property of fixed-window rate limiters. A sliding-window or
leaky-bucket implementation requires per-packet subtraction arithmetic on BPF packet pointers,
which the verifier restricts. The worst-case amplification (2× burst) is bounded and tolerable
for ICMP echo protection. A 20 pps limit allows at most 40 packets per worst-case burst, well
below any useful amplification factor.

### SEC-15 — LRU map eviction can bypass per-IP rate limiting [LOW] [ACCEPTED]

**Surface:** `ebpf/dns_xdp.c`, `icmp_rate_limit` map (65536 entries)

**Description:**
The `icmp_rate_limit` LRU map holds per-source-IP rate state. An attacker using more than
65536 distinct source IPs (e.g. via a botnet) can fill the map, causing LRU evictions. A
previously rate-limited IP that gets evicted can restart with a fresh window on its next
packet.

**Accepted risk:** 65536 source IPs exceeds the realistic scale of a single-machine attack
from a LAN. Increasing the map size linearly increases BPF locked memory. The LRU behavior
is a standard tradeoff in BPF rate limiters (Linux kernel tc/BPF uses the same pattern).
This is acceptable for protecting a single DNS server from casual abuse; it is not a
DDoS mitigation system for botnet-scale attacks.

### SEC-16 — WebUI proxy body buffer (8 MB) exceeds API backend limit [MEDIUM] [FIXED v0.9.2]

**Surface:** `src/webui/mod.rs`, `proxy_api()`, line `axum::body::to_bytes(..., 8 * 1024 * 1024)`

**Description:**
The reverse proxy reads the entire request body into a `Bytes` allocation before forwarding
to the API backend. The limit is 8 MB. The API backend enforces a smaller body limit
(`MAX_BODY_BYTES`, currently ~512 KB). A client can send up to 8 MB to the WebUI port, which
will be buffered in the proxy, then rejected 413 by the backend.

Under concurrent connections, each 8 MB body allocates a heap buffer. On a constrained VM
(512 MB RAM), ~60 concurrent large requests could saturate the heap.

The webui port binds `0.0.0.0` by default, so unauthenticated LAN clients can trigger this.

**Fix (recommended):** Align the WebUI body limit with `MAX_BODY_BYTES` from the API (expose
it as a constant and import it in `webui/mod.rs`). Secondary fix: stream the body through
reqwest without buffering (requires reqwest `Body::from(stream)` instead of `to_bytes`).

### SEC-17 — All WebUI clients share the API rate-limit bucket (127.0.0.1) [INFO] [ACCEPTED]

**Surface:** `src/webui/mod.rs` proxy forwarding, `src/api/mod.rs` rate limiter

**Description:**
The API rate limiter keys on the remote IP of the TCP connection. All requests through the
WebUI proxy arrive from `127.0.0.1`, regardless of the actual browser's IP. This means:
1. All WebUI users collectively share one rate-limit bucket
2. A single user making rapid requests through the WebUI can exhaust the bucket, causing 429
   errors for other users

**Accepted risk (v0.9.x):** Adding `X-Forwarded-For` support in the API introduces trust
chain complexity — the API must distinguish trusted proxy IPs from untrusted forwarded headers.
Given that the WebUI and API are co-hosted, this is deferred to v1.0. The WebUI is primarily
single-user (home lab / small team); multi-user rate contention is unlikely.

### SEC-18 — WebUI default bind `0.0.0.0` exposes management UI on all interfaces [INFO] [ACCEPTED]

**Surface:** `src/config/parser.rs`, `ui-bind` default; `docs/configuration.md`

**Description:**
The WebUI server binds `0.0.0.0` by default. On internet-connected machines, the management
dashboard is reachable from any IP. Authentication relies solely on the API Bearer token
(enforced by the proxied API, not by the WebUI layer itself). There is no rate limiting on
the WebUI port for failed auth attempts beyond what the API backend provides.

**Accepted risk:** The `docs/configuration.md` documents that `ui-bind: 127.0.0.1` should be
used if external access is not desired. The API key is 256-bit and not guessable by brute
force. No management function is accessible without a valid key.

**Operator note:** If the machine is internet-connected, use `ui-bind: 127.0.0.1` or protect
the UI port with a firewall rule. The default `0.0.0.0` is appropriate for LAN deployments.

## 3. Optimization Findings

### PERF-05 — WebUI proxy: full body buffering before forwarding [LOW]

**Surface:** `src/webui/mod.rs`, `axum::body::to_bytes(...)`

**Description:**
The proxy reads the entire request body into a contiguous `Bytes` before constructing the
reqwest request. For typical API use (JSON payloads <64 KB), this is negligible. For
potential future upload endpoints (zone import, feed import), this allocates the full payload
on the heap before the backend even sees the first byte.

**Recommendation:** Switch to streaming: pass the axum body stream directly to reqwest via
`reqwest::Body::from(stream)`. Reqwest supports `impl Stream<Item = Bytes>` via
`reqwest::Body::wrap_stream`. This eliminates the heap allocation and allows the backend to
begin processing while bytes are still in transit.

**Impact:** Low for current use. Worth addressing before adding bulk-import endpoints.

---

### PERF-06 — icmp_read_stats forces mutable BPF object borrow [INFO]

**Surface:** `src/dns/xdp/loader.rs`, `icmp_read_stats()`

**Description:**
`PerCpuArray::try_from(map)` requires `map_mut()`, taking `&mut self` on `XdpHandle`.
Even a read-only stats poll acquires the mutable borrow for the duration of the call.
In the current design (single poll task owns `XdpHandle`), this is not a contention issue.

**Note:** This is an aya API limitation — PerCpuArray reads require mutable map access in
aya 0.13. No action required unless XdpHandle needs concurrent access.

## 4. Items Verified Correct

The following security properties were reviewed and found to be implemented correctly:

- **BPF bounds checks:** Every pointer arithmetic result is validated against `data_end`
  before dereference. The BPF verifier enforces this; the code passes without `unsafe` tricks.
- **IP checksum preservation on MAC/IP swap:** One's complement sum is commutative — swapping
  src/dst preserves the sum. Verified by inspection and confirmed by successful ping replies
  in live testing.
- **ICMP checksum update:** `csum16_add(icmp->checksum, bpf_htons(ICMP_ECHO << 8))` correctly
  adjusts the one's complement checksum when changing type 8→0. Confirmed by live ping test
  (no "invalid checksum" messages from receiving hosts).
- **Auth enforcement on ICMP API endpoints:** All three new endpoints (`GET /icmp/stats`,
  `GET /icmp/config`, `PUT /icmp/config`) return HTTP 401 without a valid Bearer token.
  Tested against live master.
- **WebUI proxy cannot be used for SSRF:** Target is hardcoded to `http://127.0.0.1:{api_port}`.
  No user-controlled URL component. Path-and-query from the original request is appended
  verbatim, but the API's routing limits what can be reached.
- **hop-by-hop header stripping:** `host`, `transfer-encoding`, and `content-length` are
  stripped from proxied requests. `transfer-encoding` is stripped from proxied responses.
  Prevents header injection and chunked-encoding smuggling.
- **IcmpStats atomics are Relaxed-ordered:** Stats counters use `Ordering::Relaxed`. For
  monotonic counters read once per second, this is correct — no synchronization guarantee
  needed beyond the poll interval.
- **XdpHandle Drop detaches BPF program:** Verified in `loader.rs`. ICMP maps are destroyed
  on drop, preventing stale entries from persisting across restarts.
