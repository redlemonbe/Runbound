## 2. Security Findings

### SEC-A1 — MEDIUM — No brute-force protection on WebUI login

**File:** `src/webui/mod.rs`, `handle_login()` (~line 217)
**Source:** [AI-INTERNAL]

**Description:** The POST `/login` endpoint performs argon2id verification but imposes no rate limit, lockout, or delay on failed attempts. An attacker on the network can enumerate credentials at the maximum rate supported by argon2 hashing speed (typically 100–500 ms per attempt on a slow machine, faster on high-end hardware). With a 4-character minimum password and a default username of "admin", a brute-force attack is feasible.

**Reproduction:** Automated POST requests to `/login` with arbitrary passwords succeed without throttling.

**Impact:** WebUI account takeover with persistence.

**Suggested fix:** Add per-IP rate limiting in `handle_login`: DashMap keyed by IpAddr tracking failure count and window start. Lock after 5 failures per minute; apply exponential delay (1s, 2s, 4s...) on successive failures.

**Status: Open**

---

### SEC-A2 — LOW — Session cookies missing Secure flag

**File:** `src/webui/mod.rs`, lines ~247, ~252
**Source:** [AI-INTERNAL]

**Description:** Both `rb_session` and `rb_csrf` cookies are set with `HttpOnly` and `SameSite=Lax` but without the `Secure` attribute. On HTTP connections (non-TLS), both cookies are transmitted in cleartext. An on-path attacker (ARP spoofing, rogue AP) can steal session tokens from HTTP traffic.

**Context:** The WebUI runs on port 8091 with no built-in TLS. The built-in webui feature (#91) was implemented as plain HTTP. A reverse proxy is required for HTTPS.

**Suggested fix:** Add `; Secure` to both Set-Cookie headers when `X-Forwarded-Proto: https` is present, or add a `webui-force-secure` config option.

**Status: Open**

---

### SEC-A3 — LOW — Minimum WebUI password length 4 characters

**File:** `src/webui/mod.rs:298`
**Source:** [AI-INTERNAL]

**Description:** Password change validation rejects passwords shorter than 4 characters. This is too weak for a DNS server management interface where a compromised account could disrupt network resolution or exfiltrate data. The `admin` default credential bootstrapped at startup is not forced to change on first login.

**Suggested fix:** Raise minimum to 12 characters. Add a one-time startup warning if default credentials (username "admin") are still in use.

**Status: Open**

### SEC-A4 — INFO — CSRF verify_csrf() uses non-constant-time comparison

**File:** `src/webui/mod.rs:119`
**Source:** [AI-INTERNAL]

**Description:** `verify_csrf()` uses string equality (`==`) to compare CSRF tokens, not a constant-time comparator. In principle, this is a timing oracle. In practice, CSRF tokens are delivered to JavaScript via an accessible cookie (`rb_csrf`, not HttpOnly), so they are not secret in the same way passwords are. Timing-oracle exploitation requires attacker JS control on the same origin — which implies XSS, a stronger precondition.

**Impact:** Negligible in practice.

**Status: Accepted**

---

### SEC-A5 — INFO — IPv6 link-local not rejected in relay_host validation

**File:** `src/sync.rs:968-982`
**Source:** [AI-INTERNAL]

**Description:** The relay_host validation at `/nodes/register` rejects IPv4 link-local (`169.254.0.0/16`) but not IPv6 link-local (`fe80::/10`). A malicious slave could attempt to register `relay_host: "[fe80::1]:8082"`. In practice, link-local routing requires a scope ID (e.g., `%eth0`) which Rust's IpAddr parser rejects, and connections to unscoped link-local addresses fail at the OS level. SSRF via this path is not practical.

**Suggested fix:** Add explicit `(s[0] & 0xffc0) == 0xfe80` check for IPv6 addresses in the relay_host validation block, consistent with the `is_private_ip()` checks in feeds.

**Status: Accepted (impractical exploit, low priority)**

## 4. Performance Observations

**XDP stats tracking (v0.9.9):**
`stats.inc_total()` and `domain_stats.inc()` are called from XDP worker threads (OS threads). Both use `AtomicU64::fetch_add(Ordering::Relaxed)` internally — lock-free, no contention. `Relaxed` ordering is sufficient for independent counters; no happens-before relationship is needed. No correctness concerns.

**domain_stats cap (10,000 entries):**
DashMap capped at 10,000 domains prevents OOM from domain enumeration attacks. Silently ignores new entries when cap is reached. Existing entries still increment. Accepted trade-off.

**wire_qname_to_str allocation:**
`wire_qname_to_str()` in `src/dns/xdp/worker.rs` allocates a String on every XDP cache hit for `domain_stats.inc()`. At high QPS (>100k/s), this is ~100 bytes per hit under jemalloc. A future optimization is to add `domain_stats.inc_wire(&[u8])` that avoids the allocation by hashing the wire-format name directly. Not critical at current scale.

**Unsafe code survey:**
107 unsafe blocks — all in XDP/AF_XDP socket/UMEM management, CPU pinning (`sched_setaffinity`), resource limit checking (`getrlimit`), and SSE4.2 hash intrinsics (guarded by `is_x86_feature_detected!`). All are properly SAFETY-commented. No unsafe outside expected kernel-interface code.
