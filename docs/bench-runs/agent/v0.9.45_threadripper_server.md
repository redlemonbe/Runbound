# Runbound v0.9.45 — Threadripper-as-Server Benchmark

**Date:** 2026-05-26
**Tool:** dnsmark v0.4.5
**Server:** AMD Ryzen Threadripper PRO 5995WX (32 vCPUs, 15 GiB RAM) @ 192.168.8.245
**Client:** Intel Xeon E5-2690 v2 @ 3.00GHz (20 cores, 30 GiB RAM) @ 192.168.8.223
**Mode:** Userspace only — XDP/eBPF explicitly disabled (`RUNBOUND_DISABLE_XDP=1` + `--no-xdp`)
**Zone:** 100,000 A records (`bench.local`), served from local zone/local-data on each resolver
**Reference:** [v0.9.45_userspace.md](v0.9.45_userspace.md) (previous bench, roles reversed)

---

## Context: Roles Reversed

This benchmark inverts the hardware assignment of the previous run:

| Role | Previous bench | This bench |
|------|----------------|------------|
| DNS server | Intel Xeon E5-2690 v2 (20c) | AMD Threadripper PRO 5995WX (32 vCPUs) |
| dnsmark client | AMD Threadripper PRO 5995WX (32 vCPUs) | Intel Xeon E5-2690 v2 (20c) |

**Important finding:** With 4 dnsmark worker threads on the 20-core Xeon client, the client itself becomes the throughput ceiling. Phase B sustained tests top out at ~85k QPS regardless of server, and Phase C at ~120k QPS. This confirms the Threadripper server is not the bottleneck in any phase — all three resolvers reach the client-imposed ceiling simultaneously. The comparative value of this bench is latency profiles and loss rates under equivalent load, not absolute QPS.

## Benchmark Protocol

For each server:
1. All other resolvers stopped
2. Warm-up: 30s at 5,000 QPS (20 dnsmark workers, `--max-outstanding 10000`)
3. Phase A (ceiling detection): `--ramp` mode, 20 workers
4. Phase B (sustained): 60s at 80% of ceiling target, 4 workers
5. Phase C (stress): 60s at 150% of ceiling target, 4 workers
6. Memory/CPU sampled from `ps aux` at peak load

Dataset: 99,999 A-record queries against `bench.local` (100,000 unique names, seed=42).

## Analysis

**Client bottleneck in Phase B/C.** With 4 dnsmark worker threads on the 20-core Xeon client, actual throughput caps at ~85k QPS in Phase B and ~120k QPS in Phase C regardless of which server is tested. Target QPS (152k-155k and 285k-290k) exceeds what 4 client workers can generate against a fast local server. The server is never the limit.

**Peak ceiling convergence (Phase A).** With 20 ramp workers, all three servers saturate the virtual 1 GbE link between 190k and 194k QPS, within 2% of each other. This is consistent with the NIC or kernel UDP stack being the shared ceiling rather than any DNS implementation.

**Latency profiles are statistically indistinguishable.** All three resolvers deliver p50 of 0.23-0.31ms and p99 of 0.38-0.63ms in Phase B. The deltas are sub-millisecond. Under client-constrained load, no resolver shows meaningful tail latency advantage.

**Runbound p999 and max spikes.** Runbound shows occasional p999 spikes (0.635ms vs 0.478ms BIND9) and higher max latency (17.8ms vs 7.2ms BIND9 in Phase B). These are consistent with Tokio async task scheduling jitter. They do not propagate to p99 and are not operationally significant.

**Why this bench differs from the previous run (v0.9.45_userspace.md):**

1. **Server hardware (Threadripper vs Xeon):** The Threadripper serves at ~190-194k QPS ceiling vs. the Xeon server's ~55-68k QPS in Phase B of the previous bench. The Threadripper PRO with 32 vCPUs saturates the 1 GbE NIC ceiling before CPU limits are reached.

2. **Client hardware (Xeon vs Threadripper):** The Xeon client with 4 workers generates only ~85k sustained vs. the Threadripper client (20-32 workers) that pushed 140-152k QPS in Phase A of the previous bench. The weaker client limits Phase B/C comparisons.

3. **Role reversal removes differentiation.** The previous bench found Runbound leading by +21% vs BIND9 in Phase B because the Xeon server's CPU was the bottleneck and Runbound's async Tokio model handled saturation better. In this bench, the Threadripper server is never CPU-bound, eliminating that differentiation.

**XDP disabled.** All tests use `RUNBOUND_DISABLE_XDP=1` and dnsmark `--no-xdp`. These results represent the userspace-only path for all three resolvers.

## Methodology

- BIND9: authoritative-only (`recursion no`), zone loaded from file (`bench.local`)
- Unbound: `auth-zone` for `bench.local`, 32 threads, 512 MB msg+rrset cache, no upstream
- Runbound: 100,000 `local-data` entries, `rate-limit: 0`, no upstream forwarding
- Dataset: 100,000 A records, random names 4-12 chars, TLDs: com/net/org/io/co/app/dev, seed=42
- Warm-up: 30s at 5,000 QPS before each test series
- Phase B/C: 4 dnsmark client workers (`-c 4`) per protocol spec
- Network: UFW open ports 53/udp and 5353/udp, no conntrack manipulation
- RAM: RSS from `ps aux` at peak load; BIND9: 118 MB, Unbound: 168 MB, Runbound: 208 MB
