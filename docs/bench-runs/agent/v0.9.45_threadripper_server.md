# Runbound v0.9.45 — Threadripper-as-Server DNS Benchmark

**Date:** 2026-05-26
**Tool:** dnsmark v0.4.5
**Server:** AMD Ryzen Threadripper PRO 5995WX (32 vCPUs, 15 GiB RAM) @ 192.168.8.245
**Client:** Intel Xeon E5-2690 v2 (20 cores, 30 GiB RAM) @ 192.168.8.223
**Mode:** Userspace only — XDP/eBPF explicitly disabled for all resolvers (`xdp: no` + `RUNBOUND_DISABLE_XDP=1`)
**Zone:** 100,000 A records (`bench.local`), identical dataset across all resolvers
**Reference:** [v0.9.45_userspace.md](v0.9.45_userspace.md) (previous bench — Xeon server / Threadripper client)

## Analysis

**Phase A (c=10):** The Threadripper server handles low-concurrency load comfortably for all resolvers. Runbound leads at 157k QPS (+37% vs BIND9 and Unbound, which tie at ~115k QPS). However, Runbound shows higher p50 latency (1.27ms vs 0.75–0.86ms), suggesting more per-request processing overhead in userspace mode. Pass 1 for both Unbound (28k QPS) and BIND9 (84k QPS) shows a cold-start effect in the first pass; passes 2 and 3 stabilize around 155–168k QPS for both. Runbound converges faster (141k → 168k).

**Phase B (c=100):** All three resolvers converge in the 41–45k QPS range under 100-client stress — a significant drop from Phase A peaks. This points to the Xeon E5-2690 v2 client (20 cores, dnsmark) as the bottleneck: at 100 concurrent clients, the client saturates before the Threadripper server does. Runbound holds the best p50 (0.29ms) and p95 (1.69ms) under load, suggesting better queue handling under concurrency pressure compared to BIND9 (p95=2.44ms) and Unbound (p95=2.92ms).

**CPU and RAM:** The vmstat 7% aggregate across 32 vCPUs translates to roughly 2.2 vCPUs effective utilization. The Threadripper server is nowhere near saturation in either phase. Per-process top shows Runbound at ~86% of a single core, confirming single-threaded DNS path in userspace mode (no XDP, no SO_REUSEPORT workers). Unbound at 32 threads uses ~172 MiB (1.5x cache-optimized), Runbound at 206 MiB (100k local-data entries loaded in memory). BIND9 is lightest at ~100 MiB.

**Compared to previous bench (v0.9.45_userspace.md, Xeon-as-server):** With the Threadripper as server, Phase A peaks are higher for all resolvers (client Threadripper 32 vCPUs generated more load). Phase B QPS is lower here because the Xeon client saturates at 100 clients, while the Threadripper client in the previous bench could push more. This confirms the client hardware is the Phase B ceiling in this configuration.

**Bottleneck verdict:** Xeon E5-2690 v2 client (20 cores) is the Phase B goulot. The Threadripper server at 7% CPU across 32 vCPUs has headroom to spare. Phase A passes 2–3 show the Threadripper can sustain 155–168k QPS with Runbound in userspace; the client is maxed at that point (100 clients already shows regression).

## Methodology

- BIND9: recursion disabled, DNSSEC disabled, bench.local zone file (`/etc/bind/bench.local.zone`), querylog no
- Unbound: num-threads 32, so-reuseport yes, auth-zone bench.local, verbosity 0, msg-cache-size 512m
- Runbound: rate-limit 0, ui-enabled no, xdp no, 100k local-data entries, `RUNBOUND_DISABLE_XDP=1`
- Warm-up: 30s at QPS cap 5000 before each resolver's first pass
- CPU%: us+sy vmstat averaged over test window (32 vCPU server — 7% ≈ 2.2 vCPUs)
- RAM: RSS from `ps aux` at peak load
- Query file: 100k unique A records matching zone, NOERROR 100% verified before benchmark
- Network: virtio-net 1 GbE between VMs
