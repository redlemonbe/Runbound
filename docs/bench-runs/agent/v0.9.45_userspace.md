# Runbound v0.9.45 — Userspace DNS Benchmark

**Date:** 2026-05-26
**Tool:** dnsmark v0.4.5
**Client:** AMD Ryzen Threadripper PRO 5995WX (32 cores) @ 192.168.8.245
**Server:** Intel Xeon E5-2690 v2 (20 cores, 31 GB RAM) @ 192.168.8.223
**Mode:** Userspace only — XDP/eBPF explicitly disabled for all resolvers (`xdp: no` + `RUNBOUND_DISABLE_XDP=1`)
**Zone:** 100,000 A records (`bench.local`), identical dataset across all resolvers

## Analysis

**Phase A (c=10):** All three resolvers cluster tightly at 139–147k QPS. Runbound leads by ~3% over BIND9 and ~6% over Unbound. BIND9 has the lowest p50 latency (0.655 ms). Unbound shows higher p99 variance (2.549 ms) under auth-zone lookup at uneven thread load.

**Phase B (c=100):** Runbound pulls ahead at 68,319 QPS — **+21% vs BIND9** and **+31% vs Unbound**. Its Tokio async model keeps throughput high under saturation, though p95/p99 tail latencies (2.007/2.891 ms) are higher than BIND9's (0.874/1.682 ms). BIND9 delivers the most consistent tail latency under stress.

**Unbound Phase B** showed the highest variance across passes (36k–71k QPS), consistent with lock contention at high concurrency. Pass 1 logged 3,760 lost queries (0.13%), the highest drop rate across all tests.

**Memory:** Unbound is the most efficient at 72 MB RSS. BIND9 and Runbound are comparable at ~250–303 MB with 100,000 records in memory.

---

## Methodology

- BIND9: recursion disabled, DNSSEC disabled, 20 auto-detected worker threads
- Unbound: 8 threads, 512 MB msg+rrset cache, `auth-zone` (zone file) for 100k records
- Runbound: `rate-limit: 0`, `ui-enabled: no`, `xdp: no`, 100,000 `local-data` entries in config
- CPU%: us+sy from `vmstat` averaged over active benchmark window (20-core server, system-wide %)
- RAM: RSS of the DNS process at peak load (`ps aux`)
- Network: conntrack disabled, iptables flushed, tc qdisc removed on server NIC during tests
- dnsmark flag note: `-l <seconds>` used for duration (30s/60s); query file with 100k random domain names
