# Runbound v0.9.66 — XDP fast path: domain-routing ZC fix + wire-builder local-zone

**Date:** 2026-06-03
**Runbound version:** 0.9.66
**Mode:** Bare-metal AF_XDP zero-copy (Drv), Intel X520 (82599) 10 GbE
**Tool:** [dnsmark v1.2.1](https://github.com/redlemonbe/dnsmark)
**Builds on:** [v0.9.65](v0.9.65.md) (first bare-metal XDP 10 G benchmark)

> **Numbers corrected (2026-06-03):** the original version of this page reported the wire-builder gain as "+21 % throughput". That figure was measured **without the CPU governor pinned**, so it conflated the code change with frequency scaling. Re-measured with `performance` governor pinned and warm, the **reproducible** win is the **p50 latency (−33 %)**; the throughput gain is real but host-variance-bound (see below). This page now reports the honest numbers.

---

## Executive Summary

Two XDP fast-path changes shipped in v0.9.66:

1. **#155 — `xdp-domain-routing` (CPUMAP) no longer breaks zero-copy.** Enabling it on a ZC interface used to drop throughput ~40× and route onto HT siblings. `DOMAIN_ROUTING_ENABLED` moved from a frozen `volatile const` to a runtime map flag (`domain_routing_cfg`); it is now forced OFF once zero-copy is confirmed on a socket, so the XDP program falls through to the XSK ZC path. domain-routing remains available in SKB/copy mode (its cache-locality intent).

2. **#156 — hand-rolled wire builder bypasses hickory for local-zone A/AAAA answers.** `answer_dns_wire()` parses the query and builds the response straight into the UMEM frame (no `hickory_proto::Message` parse/serialize, no heap alloc). Everything else (NXDOMAIN, NODATA, wildcards, CNAME/MX/TXT, EDNS, ACL Deny, ANY, malformed) falls back to hickory — identical to previous behaviour, **no regression**.

## A/B benchmark — local-zone A answer path

Same query (`a.bench.test A`, a local static A record), X520 10 GbE, dnsmark flood. **Controlled back-to-back A/B: both binaries on the same host, `performance` governor pinned, 15 s warm-up, 5 runs each, medians.**

| | hickory (v0.9.65) | wire builder (#156) |
|---|---|---|
| **p50 latency** | **~0.52 ms** | **~0.35 ms** (**−33 %**) |
| Throughput (median) | ~5.5 M qps | ~5.9 M qps |

### What is reliable, and what is not

- **The −33 % p50 latency is the headline and the reproducible result.** hickory measures ~0.51–0.53 ms, the wire builder ~0.34–0.35 ms, on **every** run, across every session and governor state. It is mechanistically obvious: the wire path does far less work per packet (no parse, no serialize, no heap allocation), so each answer is produced faster. This is the metric to trust.

- **Throughput is host-variance-dominated and is *not* a precise figure on this rig.** The wire builder is consistently *faster* per packet, but at link saturation the throughput is bounded by RSS queue count (16 on the 82599), NIC, and co-located-VM contention on a shared host — not by the per-packet answer cost. Back-to-back A/B ratios measured anywhere from **+6 % to +21 %** depending on the session; the *same* wire binary produced 5.9 M qps in one session and 7.2 M qps 20 minutes later (other VMs/thermals moving). We therefore report throughput as a range, not a single number, and **lead with the p50 latency**.

> **Methodology lesson (this rig):** absolute throughput on a shared bare-metal host is unreliable session-to-session — even with the governor pinned. Only **back-to-back ratios within one session** are trustworthy, and **p50 latency is far more stable than throughput**. An earlier single run hit 6.48 M qps; it did not hold up as a median. Trust medians and latency, never peaks.

**NXDOMAIN / NODATA / wildcard / non-A-AAAA queries are unchanged** (hickory fallback) — a pure win on the A/AAAA answer case with zero behaviour change elsewhere.

## Methodology

- Generator: dnsmark ≥ v1.2.1 (saturates 10 G, per-packet source-port variation).
- Receiver CPU: **`performance` governor pinned** on all cores (`cpufreq scaling_governor=performance`) + 15 s warm-up before sampling. Without this, the cores down-clock mid-flood and absolute numbers are meaningless.
- Receiver NIC: `ethtool -N <nic> rx-flow-hash udp4 sdfn` (spread UDP across RSS queues) + `ethtool -A <nic> rx off tx off` (no PAUSE throttle).
- `rate-limit: 0`, local-zone static with a specific A record, `xdp: yes`.
- Correctness verified with `dig` (A NOERROR/AA, NXDOMAIN, wildcard via hickory, ACL).

## Notes & follow-up

- The CPU governor turned out to be the dominant confounder in earlier benchmarks. A `xdp-cpu-governor: performance` config option is planned so Runbound can pin the governor on its XDP cores (opt-in) — otherwise schedutil/powersave under-clocks the cores during the micro-bursty DNS hot path.
- The wire builder still called `hickory Name::read` once per packet in v0.9.66 to build the `LocalZoneSet` lookup key. **v0.9.67** removes it: `LocalZoneSet` is now keyed by the wire QNAME (CRC32c) with pre-serialised A/AAAA rdata, plus EDNS OPT echo so the fast path serves real EDNS traffic. See [v0.9.67](v0.9.67.md).
- ZC remains RSS-bound to 16 cores on the 82599 (`ethtool -X equal 40` → EINVAL). Per-core efficiency, not core count, is the lever on this NIC.
- domain-routing (CPUMAP) is intentionally gated OFF on zero-copy interfaces; it is a cache-locality feature for SKB/copy mode, not a throughput-scaling lever.
