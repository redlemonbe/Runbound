# Runbound v0.10.0 — Multi-interface XDP & dual-fibre

**Date:** 2026-06-04
**Mode:** Bare-metal AF_XDP zero-copy (Drv), 2× Intel X520 (82599) 10 GbE
**Tool:** [dnsmark v1.3.0](https://github.com/redlemonbe/dnsmark) (multi-NIC)

---

## What shipped

runbound can now run the XDP fast path on **N interfaces at once** (`xdp-interface: nic2,nic3` or `auto`). Each interface is fully independent (its own XSK sockets, workers, eBPF program — AF_XDP does not bond). This lets a single runbound instance receive across multiple 10 GbE links.

## Result — dual-fibre (2× 10 GbE)

Receiver: runbound on a Threadripper (both X520 ports). Generator: dnsmark on a dual Xeon E5-2690 v2, flooding both links. Measured **at the receiver NIC counters** (`ethtool -S`), 7 s window:

| interface | RX drained | TX responses | NIC drops |
|---|---|---|---|
| enp33s0f0 | 8.78 M | 9.28 M | 0.48 M |
| enp33s0f1 | 8.42 M | 8.91 M | 0.24 M |
| **aggregate** | **~17.2 M** | **~18.2 M** | ~0.7 M |

**~17–18 M qps aggregate** on the local-zone A answer path, at **~13 % CPU** on the receiver. Near-zero NIC drops.

### Honest caveats

- The TX figure is `tx_bytes_nic ÷ ~87 B`; the generator emitted ~16 M, so the true number sits in the **~17–18 M range** — treat it as a band, not a precise point.
- **The receiver is not the bottleneck** — 13 % CPU, near-zero drops. This is **generator-limited** (the Xeon caps ~16–17 M generating, even though it caps ~9 M *receiving*). A faster generator or 25 GbE+ would raise the number; the receiver has large headroom.
- This is the **local-zone A/AAAA fast path**, not recursive resolution under cache pressure.

## The bug that made dual-fibre work (a Rust RAII trap)

Multi-interface was silently broken: the **2nd+ interface received packets at the NIC but its workers drained 0**. Cause: each interface's `XdpHandle` owns its attached XDP program and **detaches it on `Drop`**. The non-primary handles were stored in a variable that the long-lived task did **not** capture (`async move` only captures what it references), so they were dropped right after setup → `Drop::detach()` ripped the 2nd interface's XDP program off the NIC. No program → no redirect to AF_XDP → workers polled empty sockets.

The startup log *said* `XDP program attached iface=…` (the attach succeeded, the detach came microseconds later), the workers *were* spawned and polling — so the symptom was three layers removed from the cause, and several plausible theories (TX path, XSKMAP, NUMA, completion-ring deadlock) were all dead ends.

**What cracked it:** per-interface instrumentation (counters proving `rx_drained=0` on the 2nd interface) + **kernel introspection on the live rig** — `bpftool net show` showed an XDP program on the first interface's ifindex and **none** on the second. The cause was invisible in the source; it was only visible in what the code did to the kernel.

Fix: keep the full set of `XdpHandle`s alive for the process lifetime (captured by the long-lived task, never dropped). Verified: `bpftool net show` shows a program on **every** XDP interface, and the 2nd interface's `rx_drained` went 0 → 8.4 M.

## Methodology notes

- Measure the receiver at its NIC counters, never the generator's round-trip (see [dnsmark/docs/benchmarking.md](https://github.com/redlemonbe/dnsmark/blob/main/docs/benchmarking.md)).
- `performance` governor pinned + warm-up; flow control off; RSS spread (`ethtool -N … rx-flow-hash udp4 sdfn`).
- `bpftool net show` is the ground truth for *which* XDP program is attached *where* — indispensable for multi-interface debugging.
- Per-interface debug counters live behind the `xdp-debug-counters` cargo feature (off in release).
