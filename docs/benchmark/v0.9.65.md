# Runbound v0.9.65 — DNS Performance Benchmark (first bare-metal XDP, 10 GbE)

**Date:** 2026-06-03
**Runbound version:** 0.9.65
**Mode:** **Bare-metal AF_XDP zero-copy (Drv mode)** — first real-network benchmark of the XDP hot path
**Tool:** [dnsmark v1.2.1](https://github.com/redlemonbe/dnsmark) (AF_XDP zero-copy generator, saturates 10 GbE)
**Link:** Intel X520 (82599) direct 10 GbE point-to-point

---

## Executive Summary

Runbound answers **4,772,073 QPS** over a real 10 GbE link via the AF_XDP zero-copy hot path, at **p50 1.25 ms / p99 3.72 ms**, while being flooded at **12.3 M pps** (≈2.5× its capacity). It does not crash, error, or degrade under the overload — the NIC sheds the excess in its hardware FIFO while Runbound keeps answering at full rate, CPU load average 0.53.

This is the **first time the XDP fast path has been measured at scale**: every prior benchmark (v0.9.45/46) was loopback or cross-VM, capped at ~195k QPS by the Proxmox network layer. **4.77 M QPS is ~24× that ceiling** — and it confirms the `5M+ QPS` design target of `examples/benchmark.conf` (the 82599's RSS caps distribution at 16 rings; the remaining 24 cores were idle for want of RSS queues, so a NIC with deeper RSS would go higher).

## Protocol

- **Server config:** `rate-limit: 0`, `xdp: yes`, hugepages, `cpu-affinity: yes`; a wildcard local zone (`*.bench.test → A`) answers every query locally (no recursion, no forwarding).
- **Query set:** 100,000 distinct `qNNNNNN.bench.test` names (dnsperf-format corpus).
- **Generator:** dnsmark AF_XDP flood, per-packet UDP **source-port variation** (so the server's RSS spreads the flow), `--max-outstanding 0`.
- **Both NICs:** flow control off (`ethtool -A <nic> rx off tx off`).
- **Server RSS:** UDP L4 hashing enabled (`ethtool -N nic3 rx-flow-hash udp4 sdfn`).
- **Duration:** ~14 s flood.

## Result

```
Queries sent:       152,667,140        (offered ≈ 12.3M pps — 10 GbE line rate)
Queries completed:   59,550,976        (39.0% — server is at 2.5× overload)
Throughput:           4,772,073 qps    (Runbound sustained answer rate)

Latency:
  avg:    1.321 ms
  p50:    1.251 ms
  p99:    3.719 ms
  p999:   4.065 ms

Server during flood: 16/40 cores at 74–97%, load avg 0.53, 0 crashes, 0 errors.
```

The 39% "completion" is not loss inside Runbound — it is the offered rate (12.3M) exceeding the answer capacity (4.77M); the NIC drops the surplus at the hardware ring (`rx_missed_errors`), which is the correct overload behaviour. Within capacity, completion is ~100%.

---

## The glass ceiling: 1 core → 16 cores

The first run of this benchmark showed only **448k QPS on a single core**, 39 cores idle. That was **entirely a measurement artifact**, in two layers:

1. **dnsmark sent a single 5-tuple flow** (fixed UDP source port) → the NIC's RSS hashed every packet to the same RX queue.
2. **The 82599 hashes UDP on IPs only by default** (`udp4` = IP SA/DA), ignoring L4 ports — so even varied source ports would not have helped until L4 hashing was enabled.

Fix (both required):
- dnsmark now varies the UDP source port per packet (v1.2.1).
- `ethtool -N nic3 rx-flow-hash udp4 sdfn` on the server.

Result: **1 core / 448k QPS → 16 cores / 4.77M QPS**, no code change to Runbound. The XDP datapath was never the bottleneck — Runbound binds an independent zero-copy XSK per queue and scales linearly per RSS-distributed core (~298k QPS/core here).

## Notes

- 4.77M is **RSS-bound, not Runbound-bound**: the 82599 caps RSS at 16 rings (`ethtool -X nic3 equal 40` → `Invalid argument`), leaving 24 cores idle. A NIC with deeper RSS (X710/E810/ConnectX) should scale further on the same code.
- Reproducing this requires the **receiver-side methodology**: `udp4 sdfn` RSS + flow control off (otherwise 802.3x PAUSE frames throttle the link to ~1.3M pps).
- `examples/benchmark.conf`'s anti-rebinding (`private-address: 10.0.0.0/8`) strips a local-data answer pointing at a private IP — use a public answer IP (e.g. `203.0.113.x`) + `access-control: ... allow` for a synthetic local-zone benchmark.
- The generator (dnsmark) itself saturates the 10 GbE line at 12.3M pps; it is not the limiting factor here.
