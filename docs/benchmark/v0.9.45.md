# Runbound v0.9.45 — DNS Performance Benchmark

**Date:** 2026-05-26  
**Runbound version:** 0.9.45  
**Compared against:** BIND9 9.20.23, Unbound 1.22.0  
**Mode:** Userspace only (XDP/eBPF disabled)  
**Tool:** [dnsmark v0.4.5](https://github.com/redlemonbe/dnsmark)

---

## Executive Summary

Runbound 0.9.45 delivers the highest throughput in both test configurations. Under low concurrency (10 clients), Runbound peaks at 147–157k QPS, 3–37% ahead of BIND9 and Unbound depending on server hardware. Under high concurrency (100 clients), Runbound leads at 68k QPS on the Xeon server (+21% vs BIND9, +31% vs Unbound), and holds the best tail latency (p95 1.69 ms) when the server has headroom to spare. Trade-off: Runbound's p50 latency is consistently higher than BIND9 under low load (0.9–1.3 ms vs 0.7–0.9 ms). Memory use is moderate at 206–303 MB for 100k records — heavier than BIND9 and significantly heavier than Unbound's 72 MB.

## Protocol

- **Warm-up:** 30 s at 5,000 QPS cap before each resolver's first pass
- **Phase A:** 10 concurrent clients, 30 s duration, 3 passes averaged
- **Phase B:** 100 concurrent clients, 60 s duration, 3 passes averaged
- **Query type:** UDP A records, 100k unique domain names, NOERROR expected for all
- **Passes:** 3 independent runs per phase; reported QPS and latencies are the average of the 3 passes

## Analysis

**Throughput under low concurrency (Phase A):** Runbound leads in both runs, with the margin varying by server hardware. On the Xeon (20 cores, Run 1), all three resolvers cluster at 139–147k QPS — the difference is small (5–6%). On the Threadripper (32 vCPUs, Run 2), Runbound reaches 157k QPS while BIND9 and Unbound plateau at ~115k. The Threadripper has enough headroom to expose Runbound's async-IO advantage; the Xeon does not.

**Throughput under high concurrency (Phase B):** The picture depends on which machine is the server. In Run 1 (Xeon as server), Runbound pulls clearly ahead at 68k QPS vs 56k for BIND9 and 52k for Unbound — the Tokio runtime handles connection fan-out efficiently. In Run 2 (Threadripper as server), all three resolvers converge at 42–45k QPS: the Xeon client (20 cores, dnsmark) is the bottleneck and caps the load before the server is stressed. The Threadripper server stays at 7% aggregate CPU (~2.2 effective vCPUs) across all Phase B passes.

**Tail latency:** Runbound's p50 is consistently 30–70% higher than BIND9 under low load (0.9–1.3 ms vs 0.7–0.9 ms), indicating per-query processing overhead in userspace mode where the Tokio scheduler adds measurable latency. However, under the one saturated-server scenario where the server is genuinely stressed (Run 2 Phase B), Runbound delivers better p95 (1.69 ms) than BIND9 (2.44 ms) and Unbound (2.92 ms), suggesting the async model handles backpressure more evenly.

**Unbound variance:** In Run 1 Phase B (Xeon server), Unbound passes varied between 36k and 71k QPS — a 2x swing. Pass 1 recorded 3,760 lost queries, the highest loss count across all tests. This is consistent with Unbound's reader-writer lock contention at high concurrency. The 3-pass average (52k) understates what Unbound can do on a good run and overstates what it does on a bad one.

**Memory:** Unbound is significantly lighter at 72 MB (Run 1). BIND9 uses 100–250 MB depending on configuration. Runbound uses 206–303 MB for 100k local-data entries — reasonable but not optimal for memory-constrained environments.

**Bottleneck identification:**
- Run 1 Phase B: Xeon server CPU is the ceiling (25% for BIND9, 9% for Unbound, 17% for Runbound — all below saturation but the server is clearly working harder than in Phase A).
- Run 2 Phase B: Xeon client is the ceiling. The Threadripper server at 7% has no observable constraint. QPS drops from 115–157k (Phase A) to 42–45k (Phase B) solely due to client saturation.

---

## Conclusions

1. **For high-throughput authoritative DNS on modern multi-core hardware:** Runbound userspace delivers 10–37% more QPS than BIND9 or Unbound depending on server hardware utilization. The advantage is largest when the server has headroom (Run 2 Phase A: +37%) and smallest when all resolvers are near the same ceiling.

2. **For latency-sensitive environments at low concurrency:** BIND9 has lower p50 and p95 than Runbound under light load. If p50 < 1 ms matters more than maximum QPS, BIND9 is the better choice in userspace mode.

3. **For memory-constrained deployments:** Unbound at 72 MB is the obvious choice. Runbound at 206–303 MB for 100k records is the heaviest of the three.

4. **For high-concurrency stability:** Avoid Unbound under sustained 100-client load on servers with many cores — the lock contention causes significant QPS variance across passes (36k–71k in this test). BIND9 and Runbound are both more consistent.

5. **XDP mode (not tested here):** All numbers above are for userspace-only operation. Runbound's XDP/eBPF fast path is expected to add a substantial multiplier on capable hardware. A follow-up benchmark on X540-capable hardware is pending.

## Methodology

### Resolver configurations

**BIND9 9.20.23**
- Recursion disabled, DNSSEC disabled
- Zone file: `bench.local` (100k A records, `/etc/bind/bench.local.zone`)
- Workers: auto-detected (20 threads on Xeon, 32 on Threadripper)
- `querylog no`

**Unbound 1.22.0**
- `num-threads`: 8 (Run 1) / 32 (Run 2)
- `so-reuseport: yes`
- `auth-zone: bench.local` (zone file)
- `msg-cache-size: 512m`, `rrset-cache-size: 512m`
- `verbosity: 0`

**Runbound 0.9.45**
- `rate-limit: 0`
- `ui-enabled: no`
- `xdp: no`, `RUNBOUND_DISABLE_XDP=1`
- 100,000 `local-data` entries in config

### Measurement methodology

- **CPU%:** `us+sy` from `vmstat` averaged over the active benchmark window (server-wide, not per-process)
- **RAM:** RSS of the DNS process at peak load (`ps aux`)
- **QPS / latency:** reported by dnsmark v0.4.5 per pass; 3-pass arithmetic mean for summary tables
- **Query file:** 100k unique A record names matching the zone; NOERROR rate verified at 100% before benchmarks

### Network conditions

- virtio-net 1 GbE between VMs
- `conntrack` disabled
- `iptables -F` (all chains flushed)
- `tc qdisc del` on server NIC (no artificial shaping)
